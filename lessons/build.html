Let's build each piece, one at a time.

File 1: The Database

A single table that stores every task.

The first file is schema.sql. It defines what our database looks like -- the blueprint. If you did the database tutorial, you already know this concept. A schema is a set of CREATE TABLE statements that tell the database what columns to have and what types of data they hold.

Our task manager needs exactly one table. Click each line to see what it does:

CREATE TABLE IF NOT EXISTS tasks ( id INTEGER PRIMARY KEY AUTOINCREMENT, title TEXT NOT NULL, status TEXT DEFAULT 'todo', created_at TEXT DEFAULT (datetime('now')) );

Remember from the database tutorial? The id column is the primary key -- every row gets a unique number, automatically. NOT NULL means a task must have a title -- the database will refuse to save one without it. DEFAULT means "if no value is provided, use this." So new tasks automatically start as "todo" with the current timestamp.

That's the entire database. One table, four columns, six lines of SQL. Paperclip's database has 50+ tables and 102 migration files -- but they're all just more of this.

File 2: The Server

A Node.js API that connects the database to the world.

The server is the middleman. The UI can't talk to the database directly -- it needs an API. Our server uses Express (a popular Node.js framework) and better-sqlite3 (an SQLite database library).

We'll build it step by step. Click each section to see the code:

const express = require('express'); const Database = require('better-sqlite3'); const path = require('path'); // Create app and database const app = express(); const db = new Database('tasks.db'); // Parse JSON request bodies app.use(express.json()); // Serve the UI files app.use(express.static('public')); // Create the tasks table if it doesn't exist db.exec(` CREATE TABLE IF NOT EXISTS tasks ( id INTEGER PRIMARY KEY AUTOINCREMENT, title TEXT NOT NULL, status TEXT DEFAULT 'todo', created_at TEXT DEFAULT (datetime('now')) ) `);
What's happening:

We import three things: Express (web framework), better-sqlite3 (database), and path (file paths).

express() creates our web server.

new Database('tasks.db') creates (or opens) an SQLite database file.

express.json() tells Express to parse JSON in request bodies.

express.static('public') serves our HTML file from a public/ folder.

Finally, we run the schema SQL to create the table.
// GET /api/tasks - Read all tasks app.get('/api/tasks', (req, res) => { const tasks = db .prepare('SELECT * FROM tasks ORDER BY created_at DESC') .all(); res.json(tasks); });
Remember from the API tutorial?

GET means "read data." The client is asking for all tasks.

db.prepare(...).all() runs a SQL query and returns all rows as an array.

res.json(tasks) sends the array back as JSON.

When the UI loads, the first thing it does is GET /api/tasks to fetch all existing tasks. This is the same pattern Paperclip uses: GET /api/agents returns all agents.
// POST /api/tasks - Create a new task app.post('/api/tasks', (req, res) => { const { title } = req.body; if (!title) { return res.status(400).json({ error: 'Title is required' }); } const result = db .prepare('INSERT INTO tasks (title) VALUES (?)') .run(title); const task = db .prepare('SELECT * FROM tasks WHERE id = ?') .get(result.lastInsertRowid); res.status(201).json(task); });
POST means "create."

req.body contains the JSON the client sent. We extract the title.

If no title was provided, we send back a 400 error (bad request).

INSERT INTO tasks creates a new row. The ? is a placeholder -- the actual value goes in safely via .run(title). This prevents SQL injection attacks.

Then we fetch the newly created task and return it with status 201 (created).

Same pattern as Paperclip's POST /api/issues.
// PATCH /api/tasks/:id - Update a task app.patch('/api/tasks/:id', (req, res) => { const { status } = req.body; const { id } = req.params; db .prepare('UPDATE tasks SET status = ? WHERE id = ?') .run(status, id); const task = db .prepare('SELECT * FROM tasks WHERE id = ?') .get(id); res.json(task); });
PATCH means "update part of something."

The :id in the URL is a parameter. If the client calls PATCH /api/tasks/3, then req.params.id is 3.

We extract the new status from the request body and run an UPDATE query to change just that field.

Then we fetch and return the updated task.

This is how the "mark as done" button works: the UI sends PATCH /api/tasks/3 with { status: "done" }.
// Start the server const PORT = 3000; app.listen(PORT, () => { console.log(`Server running at http://localhost:${PORT}`); });
This starts the server.

app.listen(3000) tells Express to start listening for requests on port 3000.

When you run node server.js, this line executes and the server begins accepting connections.

Open http://localhost:3000 in a browser and the server will serve your HTML page (because of the express.static middleware we set up earlier).
Paperclip comparison: Our server has 3 endpoints in one file. Paperclip's server has 43 route files with hundreds of endpoints. But the pattern is identical: define a URL path, read from or write to the database, send back JSON. The route for GET /api/agents in Paperclip looks almost exactly like our GET /api/tasks.

File 3: The UI

An HTML page that talks to the API.

The UI is a single HTML file with some JavaScript inside it. It does four things: fetch tasks when the page loads, render them as a list, let you add new tasks, and let you mark tasks as done.

Click each section to see how it works:

<!DOCTYPE html> <html> <head> <title>Task Manager</title> <style> /* styling omitted for clarity */ </style> </head> <body> <h1>My Tasks</h1> <form id="task-form"> <input id="task-input" placeholder="What needs to be done?"> <button type="submit">Add</button> </form> <ul id="task-list"></ul> <script> /* JavaScript goes here */ </script> </body> </html>
The HTML skeleton.

A heading, a form with an input and button, and an empty <ul> (unordered list) where tasks will appear.

The id attributes let our JavaScript find these elements later.

This is the same principle as Paperclip's UI: the HTML defines the structure, JavaScript fills it with data from the API.
// Fetch all tasks from the API async function loadTasks() { const response = await fetch('/api/tasks'); const tasks = await response.json(); renderTasks(tasks); } // Load tasks when the page opens loadTasks();
This is the API call.

fetch('/api/tasks') sends a GET request to the server. This is the browser calling the endpoint we defined in server.js.

response.json() parses the JSON response into a JavaScript array.

Then we pass the array to renderTasks() to display them.

Remember from the codebase tutorial? Paperclip's UI does the exact same thing: fetch('/api/agents') to load agents on the dashboard.
// Turn the tasks array into HTML function renderTasks(tasks) { const list = document .getElementById('task-list'); list.innerHTML = tasks.map(task => ` <li class="${task.status}" onclick="toggleTask(${task.id}, '${task.status}')"> ${task.title} </li> `).join(''); }
Turning data into UI.

.map() loops through every task and creates an HTML <li> for each one.

Each <li> gets a CSS class matching its status (so "done" tasks can be styled with a strikethrough).

Each <li> also gets an onclick handler that calls toggleTask() with the task's id and current status.

.join('') combines all the <li> strings into one big HTML string.

Setting list.innerHTML replaces the entire list at once.
// Handle the form submission document .getElementById('task-form') .addEventListener('submit', async (e) => { e.preventDefault(); const input = document .getElementById('task-input'); await fetch('/api/tasks', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ title: input.value }) }); input.value = ''; loadTasks(); });
Creating a new task via the API.

When the form is submitted, we prevent the default page reload with preventDefault().

Then we send a POST request to /api/tasks with the title in the body as JSON.

Notice the structure: method, headers, body -- this is the HTTP request we learned about in the API tutorial, now in actual code.

After the task is created, we clear the input and reload the full task list so the new task appears.
// Toggle a task between todo and done async function toggleTask(id, currentStatus) { const newStatus = currentStatus === 'done' ? 'todo' : 'done'; await fetch(`/api/tasks/${id}`, { method: 'PATCH', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ status: newStatus }) }); loadTasks(); }
Updating a task via the API.

If the task is currently "done," flip it to "todo." If it's "todo," flip to "done."

Send a PATCH request to /api/tasks/5 (where 5 is the task's id) with the new status.

The :id URL parameter we defined in the server knows how to extract the 5 and update only that row in the database.

Then reload the list to show the updated state.

This is the exact same flow as clicking "done" on a Paperclip issue.
The pattern is always the same: User does something in the UI → JavaScript calls the API with fetch() → Server receives the request → Server talks to the database → Server sends back JSON → JavaScript updates the page. Every web app ever built follows this loop.

How They Connect

Tracing a complete request from click to database and back.

Let's trace exactly what happens when you type "Buy groceries" and click Add. Click each step to follow the data:

User types "Buy groceries" and clicks Add | v UI (index.html) JavaScript calls fetch('/api/tasks', { method: 'POST', body: '{"title":"Buy groceries"}' }) | v SERVER (server.js) Express receives POST /api/tasks. Extracts title from req.body. | v DATABASE (tasks.db) INSERT INTO tasks (title) VALUES ('Buy groceries') Database creates row: { id: 5, title: 'Buy groceries', status: 'todo', created_at: '2026-06-17' } | v SERVER (server.js) SELECT * FROM tasks WHERE id = 5. Sends the new task back as JSON with status 201. | v UI (index.html) JavaScript calls loadTasks(). This fires GET /api/tasks. | v SCREEN UPDATES "Buy groceries" appears in the task list. The whole round trip took ~10ms.
Request lifecycle

Every interaction in a web app follows this cycle. Data flows down from the UI to the database, then back up to the screen. The server is always in the middle, acting as the gatekeeper.

This is the entire internet in one diagram. Every web app you use -- Google, Twitter, your bank -- has this same three-part structure. The UI is in your browser. The server is in a data center somewhere. The database is on a disk next to the server. They communicate over HTTP using JSON. That's it.

Add a user_id column to tasks. Each user only sees their own tasks. Remember foreign keys from the database tutorial?

Stage 3: Scale the database

7

Switch to PostgreSQL

SQLite is a single file. PostgreSQL is a real database server that can handle thousands of concurrent users. The SQL is almost identical.

8

Add migrations

Instead of one schema.sql, you write numbered migration files that change the schema over time. Paperclip has 102 of these.

9

Use an ORM

Instead of raw SQL strings, use Drizzle or Prisma -- write queries in TypeScript with autocomplete and type safety. Exactly what Paperclip does.

Stage 4: Go to production

10

Deploy

Put the server on a cloud provider (Vercel, Railway, AWS). Put the database on a managed service. Buy a domain.

11

Add real-time updates

WebSockets so that if another user adds a task, everyone sees it instantly. Paperclip uses this for the live dashboard.

12

Use a framework

Switch the UI to React or Vue. Split into components. Add routing for different pages. This is how you get to Paperclip's 726-component UI.

Here's what separates a toy app from a production app. Every single one of these is just more of the same patterns you've already seen:

Security

Authentication (who are you?), authorization (are you allowed?), input validation, rate limiting, HTTPS. Paperclip has a full middleware/ folder for this.

Error Handling

What if the database is down? What if the user sends garbage data? What if the server runs out of memory? Production apps handle every failure case.

Testing

Automated tests that verify every endpoint works. Paperclip has end-to-end Playwright tests that click through the UI and verify the results.

Logging & Monitoring

Record what happens so you can debug problems. Track response times, error rates, database queries. Paperclip logs every request.

Caching

Don't hit the database for the same data every time. Store frequently-read data in memory. Redis is a popular tool for this.

Background Jobs

Some work takes too long for a request (sending emails, processing files). Queue it up and do it in the background. Paperclip's heartbeat is a background job.

The takeaway: Production software is 10% building the feature and 90% handling everything that can go wrong. But every one of those additions is built on the same foundation: routes, database queries, and UI updates.

The mental model

Every web application in the world has three parts. Click each one to review:

The Database

What it does: Stores data permanently. Tables, rows, columns.

In our app: A single tasks table in SQLite.

In Paperclip: 50+ tables in PostgreSQL -- agents, issues, costs, approvals, companies, users...

The skill: Designing schemas, writing queries, understanding relationships between tables.

The Server

What it does: Sits between the UI and database. Receives requests, runs logic, returns responses.

In our app: 3 Express endpoints in one file.

In Paperclip: 43 route files, 130+ service files, middleware for auth and security, WebSocket server for real-time updates, adapter system for 11 agent types.

The skill: Designing APIs, handling errors, implementing business logic, managing security.

The UI

What it does: Shows data to the user and captures their input. Talks to the server via API calls.

In our app: One HTML page with inline JavaScript.

In Paperclip: 726 React components, 70+ pages, real-time WebSocket updates, complex state management.

The skill: HTML, CSS, JavaScript, component architecture, state management, API integration.
