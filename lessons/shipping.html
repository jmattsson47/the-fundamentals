Ideas are Cheap, Execution is Everything

Lesson 1 of 8 — From idea to MVP

Everyone Has Ideas

You know someone who says "I had that idea before Uber" or "I thought of that years ago." Ideas are everywhere. They cost nothing. The difference between an idea and a product is shipping it. That's what this course is about.

The single most important concept in shipping is the MVP — Minimum Viable Product. It's the smallest thing you can build that proves the idea works. Not the full vision. Not every feature. Just the core.

The MVP Mindset

Most beginners make the same mistake: they plan every feature before writing a single line of code. They design the settings page before the main feature works. They think about what version 5 looks like before shipping version 1.

The MVP approach is different. You ask: what is the one thing this product must do? Everything else gets cut — for now.

Analogy

Don't build a car. First build a skateboard. Then a scooter. Then a bike. Then a car. At every stage, the user can actually go somewhere. If you try to build a car from day one, you'll spend months building a chassis that nobody can drive.

The Key Insight

Shipping something small that works beats planning something big that never launches. You will learn more from 10 users using your MVP than from 10 months of building in isolation. The features you think matter often don't. The ones users actually need will surprise you.

Cut ruthlessly. Ship quickly. Iterate based on reality, not assumptions.

The Stack Decision

Lesson 2 of 8 — Choosing your tools

What Is a "Stack"?

Your "stack" is the combination of tools and technologies you use to build your product. It includes your programming language, framework, database, and hosting platform. Together, they form the foundation everything runs on.

Here's the truth most beginners need to hear: there is no perfect stack. There are only appropriate ones. The best stack is the one you can actually ship with.

Design Before Code

Lesson 3 of 8 — Think before you build

You Don't Need to Be a Designer

Design isn't about making things pretty. It's about deciding what the user does on each screen. Before you write any code, you need to answer: what screens exist, and what happens on each one?

This is called wireframing, and you can do it with pen and paper. Seriously. A rough sketch beats no plan at all.

The Tools

Wireframing Tools (simplest to most powerful)

Pen and paper — Fastest way to sketch ideas. Zero learning curve. Start here.

Figma — Free design tool used by professionals. Great for creating interactive mockups.

v0 by Vercel — Describe your UI in words, get code back. AI-powered prototyping.

The Critical Question

For every screen, ask: What does the user DO here? Not what it looks like. What action do they take? Every screen should have a clear purpose.

Building in Phases

Lesson 4 of 8 — Ship early, iterate often

The Four Phases

Don't try to build everything at once. Break your project into phases, and ship Phase 1 as fast as possible. Each phase builds on the last.

Phase 1: Core Feature

The one thing your app does, working end to end. No auth. No polish. Just the core feature, functional.

Timeline: 1-3 days

Phase 2: Auth + Polish

Add user accounts so data persists. Basic styling so it doesn't look broken. Error messages that help.

Timeline: 3-5 days

Phase 3: Deploy + Feedback

Put it on the internet. Share the link. Watch real people use it. Collect feedback.

Timeline: 1-2 days

Phase 4: Iterate

Fix what's broken. Add what users actually ask for (not what you assumed they wanted). Repeat.

Timeline: Ongoing

The Launch Checklist

Lesson 5 of 8 — Everything you need before going live

Before You Launch

Going live means real people will use your app. That's exciting and terrifying. This checklist covers what you should think about — but remember, not everything here is required for an MVP. The items marked "Required" are genuinely essential. Everything else can wait.

Click each item to learn what it means and check it off:

Launch Readiness: 0 / 10
  • ✓
    Error handling Required ▾

    When something goes wrong (and it will), your app should show a helpful message instead of crashing. Wrap API calls in try/catch. Show "Something went wrong" instead of a blank screen. Log errors so you can fix them later.

  • ✓
    Mobile responsiveness Required ▾

    More than half of web traffic is mobile. If your app doesn't work on a phone, you've lost most of your users. Test on a real device. Use responsive CSS. Make sure buttons are tappable and text is readable.

  • ✓
    Loading states Required ▾

    When data is loading, show a spinner or skeleton. When a button is clicked and something is processing, disable it and show feedback. Users who click a button and see nothing will click it 10 more times.

  • ✓
    HTTPS enabled Required ▾

    Your site must use HTTPS (not HTTP). Every modern hosting platform (Vercel, Netlify, Railway) gives you this for free. Without it, browsers will show a "Not Secure" warning and users will leave immediately.

  • ✓
    Environment variables Required ▾

    API keys, database URLs, and secrets must NOT be in your code. Use environment variables. Every hosting platform has a way to set these. If you push a secret to GitHub, consider it compromised.

  • ✓
    SEO basics Recommended ▾

    Add a descriptive <title> tag, a meta description, and an Open Graph image so your link looks good when shared on social media. Takes 10 minutes and makes a huge difference in how your project is perceived.

  • ✓
    Analytics Recommended ▾

    You need to know if anyone is using your app. Add a simple analytics tool like Plausible or PostHog. They tell you how many visitors you get, which pages they visit, and where they come from. Without analytics, you're flying blind.

  • ✓
    Custom domain Recommended ▾

    A custom domain (yourapp.com instead of yourapp.vercel.app) costs about $10/year and makes your project look professional. Not required for MVP, but a small investment that builds credibility.

  • ✓
    Backup strategy Optional ▾

    If you're using a managed database (Supabase, PlanetScale, Railway), backups are usually automatic. If you're self-hosting, set up daily backups. Losing user data is the fastest way to kill trust.

  • ✓
    Rate limiting Optional ▾

    Prevent abuse by limiting how many requests a user can make per minute. Not critical for MVP with 10 users, but important once you grow. Most API frameworks have middleware for this.

Key Insight

Perfect is the enemy of shipped. The "Required" items protect your users and your reputation. The "Recommended" items help you grow. The "Optional" items can wait until you have a reason to add them. Don't let this list paralyze you.

Getting Users

Lesson 6 of 8 — You built it. Now what?

The Hardest Part

"If you build it, they will come" is the biggest lie in tech. Nobody knows your product exists. Nobody is searching for it. You need to go where the people are and tell them about it.

Your goal isn't a million users. It's your first 10. Those 10 will tell you everything you need to know about what works and what doesn't.

Where to Share

Click each platform to see specific tips for getting noticed:

Hacker News

Community

Tech-savvy audience. Can drive massive traffic.

Post as "Show HN: [Your Product] — [What it does]". Be genuine. Explain what you built and why. Respond to every comment. Best posted on weekday mornings (US time).

Reddit

Community

Subreddits for every niche. Find yours.

Don't just drop a link — Reddit hates that. Share your story: what problem you solved, what you learned building it. Try r/SideProject, r/webdev, r/InternetIsBeautiful, or niche subreddits related to your product.

Twitter / X

Social Media

Build in public. Share your journey.

Post a thread about building your product: the problem, your approach, screenshots, and a link. Use the #buildinpublic hashtag. Follow and engage with other builders. Consistency matters more than any single post.

Product Hunt

Launch Platform

Dedicated platform for launching products.

Best for polished products (Phase 2+). Prepare good screenshots, a clear tagline, and a short description. Launch early in the day (midnight PT). Ask friends to upvote and leave genuine comments. One shot — make it count.

Indie Hackers

Community

Community of solo builders and bootstrappers.

Share your revenue numbers, tech stack, and challenges honestly. The community values transparency. Post milestones ($0 to first user, first paying customer, etc.). They love "how I built this" posts.

Friends & Family

Personal Network

Your first testers. Low stakes, honest feedback.

Send a direct message with the link. Ask them to try one specific thing ("can you create a task and mark it done?"). Watch them use it if possible. Their confusion is your roadmap for improvements.

The Feedback Loop

Getting users isn't a one-time event. It's a cycle. Ship, learn, improve, ship again. Every cycle makes your product better and your audience bigger.

Monitoring & Maintaining

Lesson 7 of 8 — Your app is live. Now what?

The Day After Launch

Congratulations, your app is live. But shipping is not the end — it's the beginning. Now you need to know: is it working? Are people using it? Are there errors?

This is where monitoring comes in. You need eyes on your application at all times, even when you're asleep.

The Tools

Essential Monitoring Stack

Error Tracking — Sentry
Catches JavaScript errors, API failures, and crashes. Sends you an alert when something breaks. Shows you the exact line of code that failed and what the user was doing.

Analytics — Plausible or PostHog
Plausible is simple, privacy-friendly page analytics. PostHog goes deeper with session recordings, feature flags, and funnels. Start with Plausible, upgrade to PostHog when you need more.

Uptime Monitoring — UptimeRobot or Better Stack
Pings your site every few minutes. If it goes down, you get a text or email. Free tiers available. Set it up and forget it — it'll wake you up when something is wrong.

Database Backups
Most managed databases (Supabase, PlanetScale, Railway) handle backups automatically. If you're self-hosting, set up daily backups to a separate location. Losing data is catastrophic.

Handling Bug Reports

When users report bugs, follow this process:

1. Reproduce — Can you make the bug happen on your machine? 2. Check logs — Look at Sentry for error details, check server logs 3. Fix — Write the fix, test it locally 4. Deploy — Push the fix, verify it's resolved in production 5. Follow up — Tell the user it's fixed. They'll love you for it.
Key Insight

Monitoring is not overhead — it's how you sleep at night. Set up Sentry and an uptime monitor before you share your app with anyone. It takes 15 minutes and saves you from finding out about problems from angry users.
