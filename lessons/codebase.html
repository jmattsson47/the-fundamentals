The Server

Routes, Services, Middleware — the three layers of the backend.

The server follows a clean pattern. Every request flows through three layers:

Request from browser or agent │ ▼ MIDDLEWARE Is this request allowed? Who is making it? │ ▼ ROUTE What are they asking for? Parse the request. │ ▼ SERVICE Do the actual work. Talk to the database. Return results. │ ▼ DATABASE Read or write data.
Analogy: A building with security. Middleware is the security guard at the front door — checking IDs, turning away unauthorized people. The route is the receptionist — figuring out which department you need. The service is the person in that department who actually does the work.

The Database Layer

How Paperclip talks to PostgreSQL.

The server doesn't write raw SQL. It uses an ORM (Object-Relational Mapper) called Drizzle. An ORM lets you write database queries in TypeScript instead of SQL.

Raw SQL

SELECT * FROM agents WHERE company_id = 5 AND status = 'active'

Drizzle ORM (TypeScript)

db.select() .from(agents) .where( and( eq(agents.companyId, 5), eq(agents.status, "active") ) )

They do the same thing. The ORM version gives you type safety — if you misspell a column name, TypeScript catches it before the code even runs.

Analogy: SQL is speaking the database's native language. An ORM is like a translator that lets you speak TypeScript and it converts to SQL for you. You get autocomplete, error checking, and you never have to worry about SQL syntax.

The schema lives in packages/db/src/schema/. Here's a simplified version of how a table is defined:

// packages/db/src/schema/agents.ts (simplified) import { pgTable, serial, text, integer } from "drizzle-orm/pg-core"; export const agents = pgTable("agents", { id: serial("id").primaryKey(), // auto-incrementing PK name: text("name").notNull(), // required text role: text("role"), // optional text adapter: text("adapter").notNull(), // "claude-code", "codex", etc. companyId: integer("company_id") // foreign key to companies .references(() => companies.id), });

This is the same schema concept from the database tutorial — but written in TypeScript instead of SQL. Drizzle converts it to a CREATE TABLE statement behind the scenes.

Adapters

How Paperclip talks to 11 different AI agents the same way.

Paperclip supports Claude Code, Codex, Cursor, Gemini, and more. But each one works differently — different APIs, different protocols, different ways to start and stop. The adapter pattern solves this.

Analogy: Think of a power outlet. Different countries have different plug shapes. An adapter lets you plug any device into any outlet. Paperclip's adapters let any AI agent plug into the same system — the server doesn't care what's on the other end.

Every adapter implements the same interface — a contract that says "you must support these operations":

// What every adapter must be able to do (simplified): interface AgentAdapter { start(task: Task, context: RunContext): Promise<RunHandle>; stop(handle: RunHandle): Promise<void>; isAlive(handle: RunHandle): Promise<boolean>; getOutput(handle: RunHandle): Promise<RunOutput>; }

Click an adapter to see how it works:

claude-local adapter

How it works: Spawns a Claude Code CLI process on the local machine. Passes the task as a prompt. Reads stdout for output. Monitors the process for completion.

Location: packages/adapters/claude-local/

This is what you're using right now — Claude Code running as a terminal process.

codex-local adapter

How it works: Spawns an OpenAI Codex CLI process. Similar to Claude — local process, stdin/stdout communication.

Location: packages/adapters/codex-local/

cursor-local adapter

How it works: Launches Cursor in headless mode. Can run locally or in the cloud. Communicates via Cursor's API.

Location: packages/adapters/cursor-local/ and cursor-cloud/

openclaw-gateway adapter

How it works: Sends HTTP requests to an OpenClaw instance running elsewhere. This is a remote agent — it doesn't run on your machine.

Location: packages/adapters/openclaw-gateway/

This is the "webhook bot" model — Paperclip sends a message, OpenClaw does the work, sends results back.

bash/CLI adapter

How it works: Runs any shell command or script. The simplest adapter — just execute a command and capture output.

This is how you'd plug in any CLI tool — a Python script, a custom agent, anything that runs in a terminal.

The beauty: the heartbeat engine doesn't know or care which adapter it's using. It just calls adapter.start(task) and the right thing happens. Add a new AI agent? Write one adapter file. Everything else just works.

The Heartbeat Engine

The most complex piece — 11,573 lines that make agents autonomous.

This is the part that makes Paperclip special. Without it, agents just sit there. The heartbeat engine wakes agents up on a schedule, checks if they have work, gives them a task, monitors execution, and handles everything that can go wrong.

Analogy: The heartbeat engine is like a factory floor manager. Every 30 seconds, they walk the floor: "Claude, you have a new task. Codex, are you still working? Scout, your budget is used up — stop. Pixel, your task is blocked — skip it." They never stop walking.

Here's what happens on every heartbeat cycle:

1

Wake Up

The scheduler fires every 30 seconds (configurable). It checks the database for agents due for a heartbeat.

2

Budget Check

Before doing anything, check: has this agent exceeded their monthly budget? If yes, skip them. No work allowed.

3

Task Checkout

Find the highest-priority task assigned to this agent. Atomically check it out — meaning no other agent can grab it at the same time. This prevents double-work.

4

Context Assembly

Build the full context: task details, goal ancestry ("why are we doing this"), project info, workspace path, secrets, skills. Everything the agent needs.

5

Adapter Invocation

Call adapter.start(task, context). This spawns the agent process (or sends the HTTP request). The engine gets back a run handle.

6

Monitoring

Track the run: Is it still alive? How long has it been running? How many tokens has it used? Is it stuck?

7

Completion

When the agent finishes: record costs, save output, update task status, log activity, release the checkout lock, and emit events for the dashboard.

8

Error Recovery

If a run crashes, times out, or orphans (server restart mid-run), detect it and clean up. Reschedule if appropriate.

The reason this file is 11,573 lines: every one of those steps has edge cases. What if two heartbeats fire simultaneously? What if the agent crashes mid-task? What if the budget runs out mid-run? What if the server restarts while an agent is working? Each edge case is handled.
