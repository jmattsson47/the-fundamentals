The Problem TypeScript Solves

Lesson 1 of 8 — JavaScript lets you do anything, including things that break

JavaScript is Too Forgiving

JavaScript will happily let you write code that looks fine but crashes when it runs. It never warns you. It just... breaks. In production. At 2am. When a user is trying to pay for something.

Here are three real bugs that JavaScript is completely fine with:

Bug #1: Wrong argument type

function calculateTotal(price, quantity) { return price * quantity; } // Oops — passing a string instead of a number calculateTotal("29.99", 3); // Returns 89.97... but only by luck. // What about: calculateTotal("twenty", 3); // Returns NaN — the order silently fails

Bug #2: Property that doesn't exist

const user = { name: "Jared", email: "jared@example.com" }; // Typo: "naem" instead of "name" console.log(user.naem); // Returns undefined — no error, no warning // Your UI just shows "Hello, undefined!"

Bug #3: Missing function argument

function greetUser(firstName, lastName) { return `Hello, ${firstName} ${lastName}!`; } greetUser("Jared"); // Returns "Hello, Jared undefined!" // JavaScript doesn't care that you forgot an argument

What TypeScript Actually Is

TypeScript is JavaScript with types. That's it. You write the same code you already know, but you add annotations that tell TypeScript what kind of data each variable, parameter, and return value should be.

TypeScript checks your code before it runs. If something doesn't match, it tells you immediately — in your editor, with a red underline, before you even save the file.

JavaScript

function add(a, b) { return a + b; } // No idea what a and b are // Strings? Numbers? Objects? // JavaScript says: "whatever"

TypeScript

function add(a: number, b: number): number { return a + b; } // Crystal clear: numbers in, number out // Pass a string? Instant error.
Analogy

JavaScript is like building with no blueprints. You can nail boards together however you want. Sometimes it works. Sometimes the roof collapses. TypeScript is adding blueprints. Before you build, you check that everything fits. The house still gets built the same way — but now you catch problems before the walls go up.

Key Insight

TypeScript doesn't run in the browser. It gets compiled (converted) to plain JavaScript before it runs. The types are only there for you, the developer. They disappear completely in the final code.

// You write this (TypeScript): const age: number = 25; // Browser runs this (JavaScript): const age = 25;

The : number annotation is stripped away. Zero runtime cost. All the safety, none of the overhead.

Types — The Basics

Lesson 2 of 8 — string, number, boolean, and arrays

The Four Core Types

TypeScript has the same data types as JavaScript. The difference is that you declare which type a variable holds. Click each type to see how it works:

Variable Annotations

To tell TypeScript what type a variable is, you add : type after the variable name:

// Explicit type annotations let username: string = "Jared"; let age: number = 28; let isActive: boolean = true; let scores: number[] = [95, 87, 92]; // Now TypeScript protects you: username = 42; // Error: Type 'number' is not assignable to type 'string' age = "twenty-eight"; // Error: Type 'string' is not assignable to type 'number' isActive = "yes"; // Error: Type 'string' is not assignable to type 'boolean' scores.push("A+"); // Error: Argument of type 'string' is not assignable

Function Parameters

The real power shows up in function parameters. In JavaScript, you have to hope callers pass the right types. In TypeScript, you enforce it:

JavaScript

function sendEmail(to, subject) { // What if `to` is a number? // What if `subject` is undefined? // We just hope for the best... }

TypeScript

function sendEmail( to: string, subject: string ) { // TypeScript guarantees both // are strings. No hoping. }

Type Inference

TypeScript is smart. If you assign a value immediately, TypeScript infers the type. You don't always need to write it:

// TypeScript infers these types automatically: let name = "Jared"; // TypeScript knows this is a string let count = 42; // TypeScript knows this is a number let items = [1, 2, 3]; // TypeScript knows this is number[] // You still get the same protection: name = 100; // Error! TypeScript inferred `name` is a string
Analogy

Types are like labeled containers in a kitchen. The sugar jar is labeled "Sugar" — you only put sugar in it. The flour canister is labeled "Flour." Without labels, someone puts salt in the sugar jar and every recipe fails. Types are the labels. They don't change what's inside — they just make sure the right thing goes in the right place.

When to Add Type Annotations

  • Function parameters — always annotate. TypeScript can't infer what callers will pass.
  • Variables with assignment — usually let TypeScript infer. let x = 5 is fine.
  • Variables without assignment — annotate. let x: number; because TypeScript doesn't know what's coming.
  • Return types — optional but helpful. We'll cover these in Lesson 4.

Interfaces & Objects

Lesson 3 of 8 — Define the shape of an object

The Problem with Plain Objects

In JavaScript, objects can have any shape. You can add or forget any property and nothing stops you. But in a real app, objects need to follow a consistent structure. An agent in Paperclip must always have an id, a name, a role, and an adapter.

Defining an Interface

An interface tells TypeScript: "Any object of this type must have exactly these fields, with exactly these types."

interface Agent { id: number; name: string; role: string; adapter: string; }

Now any object labeled as an Agent must have all four fields:

// This works — all fields present const agent: Agent = { id: 1, name: "Research Assistant", role: "researcher", adapter: "openai" }; // This fails — missing 'adapter' const badAgent: Agent = { id: 2, name: "Writer", role: "writer" }; // Error: Property 'adapter' is missing in type

Optional Properties

Sometimes a field is optional. Use ? to mark it:

interface Agent { id: number; name: string; role: string; adapter: string; description?: string; // optional — can be omitted maxTokens?: number; // optional } // Both of these are valid: const a: Agent = { id: 1, name: "Bot", role: "helper", adapter: "openai" }; const b: Agent = { id: 2, name: "Bot", role: "helper", adapter: "openai", description: "A helpful bot" };
Analogy

An interface is a form template. A tax form says: "Name (required), SSN (required), Middle Name (optional)." You can leave optional fields blank, but every required field must be filled in. If you submit a form missing a required field, it gets rejected. That's exactly what TypeScript does with interfaces.

Interfaces vs Types

You'll also see type used instead of interface. They're very similar:

// These do almost the same thing: interface Agent { id: number; name: string; } type Agent = { id: number; name: string; };

The convention: use interface for object shapes, use type for everything else (unions, intersections, aliases). You'll see both in real codebases.

Functions & Return Types

Lesson 4 of 8 — Type the inputs AND the output

Typing Function Parameters

You already know how to type parameters from Lesson 2. But functions also have a return type — the type of value they send back:

function getAgentsByCompany(companyId: number): Agent[] { // Must return an array of Agent objects return db.query("SELECT * FROM agents WHERE company_id = ?", [companyId]); }

The : Agent[] after the parentheses is the return type. It tells TypeScript (and every developer reading this code): "This function will return an array of Agents. Guaranteed."

Why Return Types Matter

Without return types, you can accidentally return the wrong thing and nobody notices until the app breaks:

JavaScript (no protection)

function getTotal(items) { if (items.length === 0) { return "No items"; // string! } return items.reduce( (sum, i) => sum + i.price, 0 ); // number! } // Sometimes returns string, // sometimes number. Chaos.

TypeScript (protected)

function getTotal( items: Item[] ): number { if (items.length === 0) { return "No items"; // Error: Type 'string' is // not assignable to 'number' } return items.reduce( (sum, i) => sum + i.price, 0 ); }

Real Codebase Example

In Paperclip, functions are typed end-to-end. Here's what a route handler looks like with full types:

interface Agent { id: number; name: string; role: string; adapter: string; } async function getAgents(companyId: number): Promise<Agent[]> { const agents = await db.select() .from(agentsTable) .where(eq(agentsTable.companyId, companyId)); return agents; // TypeScript verifies this matches Agent[] } // When you call it, TypeScript knows exactly what you get back: const agents = await getAgents(42); agents[0].name; // TypeScript knows this is a string agents[0].salary; // Error: Property 'salary' does not exist on type 'Agent'

Void and Never

Two special return types:

// void: function doesn't return anything function logMessage(msg: string): void { console.log(msg); // no return statement needed } // never: function never finishes (throws or infinite loop) function throwError(msg: string): never { throw new Error(msg); }

Generics

Lesson 5 of 8 — The hard concept, made simple

The Problem

Sometimes you want a function that works with any type, but you still want TypeScript to keep track of which type it's using. Consider this:

// This function returns the first item of an array. // But what TYPE is that item? function getFirst(items: any[]): any { return items[0]; } const name = getFirst(["Alice", "Bob"]); // name is `any` — TypeScript has no idea it's a string // name.toUpperCase() — no autocomplete, no safety

Using any defeats the purpose of TypeScript. You want a function that works with any type but remembers which type it's using.

The Solution: Generics

A generic is a type variable. You put it in angle brackets <T> and it acts as a placeholder that gets filled in when you call the function:

function getFirst<T>(items: T[]): T { return items[0]; } // Now TypeScript tracks the type: const name = getFirst(["Alice", "Bob"]); // name is `string` — TypeScript figured it out! name.toUpperCase(); // works, autocomplete and everything const age = getFirst([25, 30, 35]); // age is `number` age.toFixed(2); // works

Generics You Already Use

You've been using generics without knowing it. Every time you see angle brackets, that's a generic:

// Arrays are generic const names: Array<string> = ["Alice", "Bob"]; // Same as: string[] // Promises are generic const response: Promise<Agent> = fetchAgent(1); // "This promise will resolve to an Agent" // API responses are generic const result: Response<Agent[]> = await api.get("/agents"); // "The response body will be an array of Agents"
Analogy

A generic is a shipping box. The box itself can hold anything — books, electronics, food. But every box has a label that says what's inside this specific box. Box<Books> tells you this box has books. Box<Electronics> has electronics. Same box design, different contents, and the label keeps you from opening a box of dishes expecting books.

Writing Your Own Generic

Here's a practical example — a function that wraps an API response:

interface ApiResponse<T> { data: T; success: boolean; timestamp: number; } // When you use it, T gets replaced: const agentResponse: ApiResponse<Agent> = { data: { id: 1, name: "Bot", role: "helper", adapter: "openai" }, success: true, timestamp: 1718700000 }; // TypeScript knows agentResponse.data is an Agent agentResponse.data.name; // string ✓ agentResponse.data.foo; // Error: Property 'foo' does not exist

Union Types & Type Guards

Lesson 6 of 8 — When a value could be one of several types

What is a Union Type?

Sometimes a value genuinely could be more than one type. A user ID might be a string or a number. An API response is either a success or an error. The | operator lets you say "this OR that":

// This variable can be a string OR a number let id: string | number; id = "abc-123"; // valid id = 42; // also valid id = true; // Error: Type 'boolean' is not assignable to 'string | number'

The Problem with Unions

When a value could be multiple types, TypeScript only lets you use methods that exist on all the types:

function formatId(id: string | number) { id.toUpperCase(); // Error: Property 'toUpperCase' does not exist on type 'number' // TypeScript says: "This MIGHT be a number. // Numbers don't have toUpperCase. I can't let you do this." }

Type Guards: Narrowing the Type

You narrow a union type by checking what it is at runtime. TypeScript is smart enough to follow your if statements:

function formatId(id: string | number): string { if (typeof id === "string") { // Inside this block, TypeScript KNOWS id is a string return id.toUpperCase(); // works! } else { // Here, TypeScript KNOWS id is a number return id.toString(); // works! } }

Real Example: API Responses

This pattern is everywhere in real codebases. You make an API call and check whether it succeeded:

type Result<T> = | { ok: true; data: T } | { ok: false; error: string }; async function loadAgents(): Promise<void> { const result: Result<Agent[]> = await api.get("/agents"); if (result.ok) { // TypeScript knows: result.data is Agent[] console.log(`Loaded ${result.data.length} agents`); result.data.forEach(a => console.log(a.name)); } else { // TypeScript knows: result.error is string console.error(`Failed: ${result.error}`); } }
Analogy

A union type is like a package at your door. It might be the book you ordered or the return label you requested. Before you open it, you don't know which. But once you check the label (type guard), you know exactly what's inside and how to use it.

TypeScript in the Codebase

Lesson 7 of 8 — Real patterns from Paperclip

Types Flow Through Everything

In the Paperclip codebase, TypeScript isn't just sprinkled on top — it flows through every layer. From the database schema, through the API routes, to the frontend components. This is where the real power shows up.

Pattern 1: Typed Routes

Every API route in Paperclip has typed request and response bodies. If someone changes the database schema, TypeScript immediately shows every route that needs to be updated:

// Route definition with typed parameters and response app.get("/api/agents/:companyId", async (req, res) => { const companyId: number = parseInt(req.params.companyId); const agents: Agent[] = await db .select() .from(agentsTable) .where(eq(agentsTable.companyId, companyId)); // TypeScript verifies this response matches what the frontend expects res.json({ agents, count: agents.length }); });

Pattern 2: Database Schema Types

The database schema defines the types once, and they flow through the entire app. This is connected to what you learned in Course 01 (Databases):

// Schema definition (Drizzle ORM) export const agents = pgTable("agents", { id: serial("id").primaryKey(), name: text("name").notNull(), role: text("role").notNull(), adapter: text("adapter").notNull(), companyId: integer("company_id").references(() => companies.id), }); // TypeScript automatically generates the type: type Agent = typeof agents.$inferSelect; // { id: number; name: string; role: string; adapter: string; companyId: number | null } // Now EVERY function that touches agents uses the same type. // Change the schema? TypeScript tells you everywhere to update.

Pattern 3: Adapter Interfaces

Paperclip supports multiple AI providers (OpenAI, Anthropic, etc). Each adapter must implement the same interface:

interface AIAdapter { name: string; sendMessage(prompt: string, options: AdapterOptions): Promise<string>; streamMessage(prompt: string, options: AdapterOptions): AsyncGenerator<string>; } // Every adapter must match this shape: const openaiAdapter: AIAdapter = { name: "openai", async sendMessage(prompt, options) { /* ... */ }, async *streamMessage(prompt, options) { /* ... */ }, }; // Forget to implement streamMessage? TypeScript won't let you. // Wrong return type? TypeScript catches it instantly.

The Type System as Documentation

Here's the key insight that connects to Course 02 (The Codebase): TypeScript types serve as documentation that never goes stale.

A comment might say "this returns an array of agents" but the code might have changed. A TypeScript type is the code — if it says Agent[], that's what it returns. Period. If someone changes it and doesn't update the callers, the build breaks. You can't have outdated type documentation because the compiler enforces it.

Getting Started

Lesson 8 of 8 — Setting up TypeScript in a real project

tsconfig.json

Every TypeScript project has a tsconfig.json file at the root. It tells the compiler how to behave — how strict to be, where to find files, what version of JavaScript to output:

{ "compilerOptions": { "target": "ES2022", // What JS version to output "module": "commonjs", // Module system "strict": true, // Enable all strict checks "outDir": "./dist", // Where compiled JS goes "rootDir": "./src", // Where TS source files live "esModuleInterop": true, // Better import compatibility "skipLibCheck": true, // Skip checking node_modules "forceConsistentCasingInFileNames": true }, "include": ["src/**/*"], // Which files to compile "exclude": ["node_modules"] // Which files to ignore }

Compiling with tsc

The TypeScript compiler (tsc) converts your .ts files into .js files that the browser or Node.js can run:

# Install TypeScript $ npm install typescript --save-dev # Compile all .ts files to .js $ npx tsc # Watch mode: recompile whenever you save $ npx tsc --watch

The .ts to .js Relationship

Every .ts file compiles to a corresponding .js file. The types are stripped away:

src/agent.ts (you write)

interface Agent { id: number; name: string; } function greet(agent: Agent): string { return `Hello, ${agent.name}`; }

dist/agent.js (compiler output)

// Interface is gone completely function greet(agent) { return `Hello, ${agent.name}`; }

Notice: the interface is completely removed. The type annotation on the parameter is removed. The return type is removed. What's left is plain JavaScript. The types only exist at compile time.

How Paperclip Uses tsx

Instead of compiling with tsc and then running with node, Paperclip uses tsx — a tool that compiles and runs TypeScript in one step:

# Instead of this two-step process: $ npx tsc # step 1: compile .ts to .js $ node dist/server.js # step 2: run the .js # Paperclip uses this one-step process: $ npx tsx src/server.ts # compile + run in one step # In package.json, you'll see: "scripts": { "dev": "tsx watch src/server.ts" }

Practical Patterns

Start Gradually

You don't have to type everything at once. TypeScript supports .js files alongside .ts files. You can migrate one file at a time. Just rename server.js to server.ts and start adding types.

Use "strict" Mode

In tsconfig.json, set "strict": true. This enables the strictest checks. It catches more bugs. Yes, it means more red underlines at first, but every one of those is a potential runtime bug you're preventing.

Let TypeScript Infer When It Can

Don't annotate everything. If TypeScript can figure out the type, let it. Annotate function parameters (TypeScript can't infer those) and complex return types. Let inference handle the rest.

What You've Learned

Summary

TypeScript adds a safety layer to JavaScript that catches bugs before your code runs. Types, interfaces, generics, and unions give you a precise vocabulary to describe your data. The compiler checks everything at build time and then strips the types away, leaving clean JavaScript. In a real codebase like Paperclip, types flow from the database through the API to the frontend — acting as documentation that the compiler enforces.
