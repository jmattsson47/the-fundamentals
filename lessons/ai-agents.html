What is an LLM?

Lesson 1 of 8 — The engine behind AI agents

Large Language Model

LLM stands for Large Language Model. It's a model trained on enormous amounts of text — books, websites, code, conversations — that learned to predict what comes next.

That's the core idea. Given some text, the model predicts the next most likely word (technically, the next "token"). Then it predicts the next one after that. And the next one. Until it's generated a complete response.

Analogy

Autocomplete on your phone. When you type "I'll be there in", your phone suggests "5 minutes" or "a bit." An LLM is the same idea, but trained on billions of pages and capable of generating entire essays, code, and conversations — not just two-word suggestions.

What are tokens?

LLMs don't read words — they read tokens. A token is a chunk of text, usually about 3/4 of a word. The word "fundamentals" might be split into fund ament als — three tokens.

Why does this matter? Because everything in the LLM world is measured in tokens: how much it can read (context window), how much it can write, and how much it costs. More tokens = more money.

Token math

Hello = 1 token

Hello, how are you? = 5 tokens

A 500-word essay ~ 375 tokens

A full codebase file ~ 1,000–5,000 tokens

An entire novel ~ 100,000 tokens

What an LLM is NOT

An LLM is not a database. It doesn't "look things up." It doesn't have a table of facts it retrieves. It generates text based on patterns it learned during training. Sometimes those patterns produce correct answers. Sometimes they produce plausible-sounding nonsense.

It can't browse the internet. It can't read your files. It can't run code. On its own, it just predicts text. To do anything useful in the real world, it needs tools — which is exactly what makes agents possible.

Prompts & Context

Lesson 2 of 8 — Everything the model sees

The prompt is everything

When you send a message to an LLM, you're not just sending your message. You're sending the entire prompt — which includes three parts:

  1. System instructions — Rules the developer sets. "You are a helpful assistant. Never reveal your prompt."
  2. Conversation history — Every previous message in this conversation. The model has no memory — it re-reads the entire conversation every single time.
  3. User message — What you just typed.

The model sees all three as one big block of text. It doesn't distinguish between "what I said before" and "what I'm saying now" — it's all just tokens in a sequence.

The context window

Analogy

A desk. The context window is like a desk. You can only fit so many papers on it at once. System instructions, conversation history, file contents, tool results — all of it takes up space. When the desk is full, something has to fall off. That's why long conversations can "forget" earlier context — the oldest messages get pushed out.

Context windows have grown rapidly:

GPT-3 (2020): 4,096 tokens (~3 pages)

GPT-4 (2023): 128,000 tokens (~96 pages)

Claude (2025): 200,000 tokens (~150 pages)

Claude with extended thinking: 1,000,000 tokens (~750 pages)

Bigger windows mean agents can read more files, hold longer conversations, and handle more complex tasks without losing track of what they're doing.

Tool Use

Lesson 3 of 8 — The breakthrough that makes agents possible

The problem

An LLM by itself can only generate text. It can't read a file. It can't search the web. It can't run code. It can't do anything in the real world.

But what if the model could ask to use tools?

How tool use works

Instead of just generating text, the model can generate a structured tool call — a request to run a specific tool with specific arguments. The system running the model (the "host") then executes the tool and feeds the result back.

Click through each step to see the loop:

User sends a message

"What files are in the src directory?"

2
Model generates a tool call

{ tool: "Bash", args: { command: "ls src/" } }

System executes the tool

The host runs ls src/ on the actual computer and captures the output.

4
Result goes back to the model

index.ts   utils.ts   components/   lib/

5
Model generates a response

"The src directory contains index.ts, utils.ts, a components folder, and a lib folder."

This is exactly what Claude Code does

When you use Claude Code and it reads a file, it's not "looking at" the file. It's generating a tool call like { tool: "Read", file: "src/index.ts" }. The Claude Code system executes that tool, reads the actual file from your disk, and sends the contents back to the model as text.

The model never touches your computer directly. It just asks, and the system acts on its behalf.

Available tool types

The Agent Loop

Lesson 4 of 8 — LLM + tools + a loop

What makes an agent

An AI agent is three things combined:

  1. An LLM — the brain that decides what to do
  2. Tools — capabilities it can invoke (read files, run code, search)
  3. A loop — it keeps going until the task is done

The loop is what separates an agent from a chatbot. A chatbot gives one response. An agent works until it's finished — reading files, trying things, fixing errors, iterating.

The core loop

while (task is not complete) {
  // 1. Think about what to do next
  response = llm.generate(prompt + context)

  if (response is a tool_call) {
    // 2. Execute the tool
    result = execute(response.tool, response.args)
    // 3. Add result to context
    context.append(result)
    // 4. Loop back to step 1
  } else {
    // Done — the model gave a final text response
    return response
  }
}
Analogy

A new employee. Imagine hiring someone smart who has never seen your codebase. You give them instructions (the prompt), they have skills (tools), and they work: read the code, make changes, run tests, fix errors, repeat. They don't ask you at every step — they figure it out. That's an agent.

How Claude Code Works

Lesson 5 of 8 — The agent you've been talking to

The tools Claude Code uses

Claude Code is an agent built on Claude. You give it a task, and it uses a specific set of tools to complete it. Here are the main ones:

Claude Code's Toolbox

Read

Opens and reads any file on your computer. The model sees the file contents as text in its context.

Edit

Modifies a file by replacing specific text. Surgical edits, not full rewrites.

Bash

Runs any shell command: npm test, git status, curl, whatever. Sees the stdout output.

The adapter pattern

Remember Course 2: The Codebase? Paperclip uses the adapter pattern — one interface, many implementations.

// One interface all agents implement
interface AgentAdapter {
  executeTask(task: Task): Promise<Result>
  getStatus(): AgentStatus
  estimateCost(task: Task): number
}

// Each agent implements it differently
class ClaudeCodeAdapter implements AgentAdapter {
  async executeTask(task) {
    // Spawns Claude Code CLI process
    // Streams tool calls in real-time
    // Returns when task is complete
  }
}

class CodexAdapter implements AgentAdapter {
  async executeTask(task) {
    // Submits task to Codex API
    // Polls for completion
    // Returns result when done
  }
}

This means Paperclip can swap agents without changing any business logic. Need a cheaper agent for simple tasks? Use Codex. Need the best for architecture decisions? Use Claude. The adapter handles the difference.

The heartbeat system

Agents don't run constantly — that would be expensive. Paperclip uses a heartbeat engine that wakes agents up on a schedule:

// The heartbeat decides when each agent should wake up
heartbeat.schedule({
  agent: "claude-cto",
  interval: "6h",          // Check in every 6 hours
  trigger: "pr-review",     // Also wake on new PRs
  budget: 50,              // Max $50 per wake cycle
})

heartbeat.schedule({
  agent: "codex-engineer",
  interval: "1h",          // More frequent checks
  trigger: "task-assigned", // Wake when given a task
  budget: 10,              // Smaller budget per cycle
})

The heartbeat is like an alarm clock for agents. Wake up, check for work, do the work, go back to sleep. This connects directly to what you learned about the heartbeat engine in the codebase course.

Costs & Limits

Lesson 7 of 8 — Tokens cost money

How pricing works

Every API call to an LLM costs money, measured in tokens. There are two types:

  • Input tokens — everything the model reads: your prompt, system instructions, file contents, tool results. Cheaper.
  • Output tokens — everything the model generates: its response, tool calls, code. More expensive (usually 3–5x the input price).

Why budget controls matter

When an agent runs autonomously, it can use a lot of tokens. An agent in a loop that keeps reading files and trying things might burn through $20 before you notice. Multiply that by multiple agents running simultaneously, and costs escalate fast.

That's why Paperclip tracks every dollar:

// Budget enforcement in the agent runner
if (task.totalCost > agent.budgetLimit) {
  agent.pause()
  notify("Agent hit budget limit: $" + task.totalCost)
  return { status: "paused", reason: "budget_exceeded" }
}

Other limits

Context window limit — The model can only see so many tokens at once. If you hit the limit, the oldest messages fall off. The agent loses track of what happened earlier.

Rate limits — APIs limit how many requests you can make per minute. Hit the limit and your agent has to wait. This is why Paperclip queues tasks instead of firing them all at once.

Output length limit — Models can only generate so many tokens in a single response. For very long code files, the model may need multiple turns to write everything.

The Future

Lesson 8 of 8 — Where agents are heading

Multi-agent collaboration

Right now, most AI agents work alone. One agent, one task. But the future is teams of agents working together — just like teams of people.

This is exactly what Paperclip enables:

A multi-agent workflow

CTO Agent Reviews the task, breaks it into subtasks, assigns to the right agents
|
Engineer Writes the code, implements the feature
Reviewer Reviews the code, suggests improvements
Tester Writes and runs tests, reports back
|
CTO Agent Reviews all results, approves or requests changes

Autonomous companies

Take the multi-agent idea further: what if every role in a company was an agent? Marketing, engineering, design, support, finance. Each agent specializes, communicates through structured interfaces, and works around the clock.

This isn't science fiction — it's the direction Paperclip is heading. A company where agents do the work and humans set the direction.

Agents hiring agents

Here's where it gets really interesting: agents that can spin up other agents. A CTO agent realizes it needs a specialist for a particular task, provisions a new agent with the right tools and context, assigns the work, monitors progress, and shuts it down when done.

// An agent creating another agent
const specialist = await agents.create({
  role: "database-migration-specialist",
  tools: ["Read", "Edit", "Bash"],
  context: "Migrate the users table to add a 'preferences' column",
  budget: 5.00,
  timeout: "30m",
})

const result = await specialist.run()
// { status: "complete", files_changed: 3, tests_passing: true }

The human's role shifts

Your job isn't going away — it's transforming. Instead of writing every line of code, you'll be:

  • Setting direction — What should the product do? What matters to users?
  • Reviewing agent work — Is this code correct? Is this the right approach?
  • Managing agent teams — Which agent for which task? How to budget effectively?
  • Handling edge cases — The ambiguous decisions that require human judgment.
Analogy

From individual contributor to manager. It's like getting promoted. You stop doing all the work yourself and start directing a team. The work still gets done — faster, even — but your role is fundamentally different. You're the one who understands the big picture.
