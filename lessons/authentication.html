The Problem

Lesson 1 of 8 — Why does a server need to know who you are?

A World Without Auth

Imagine you build a beautiful web app. Users can create accounts, post messages, and manage their settings. You deploy it. Everything works. Then someone hits your API directly:

# Anyone can do this. No login required.
DELETE /api/users/42
POST   /api/admin/make-admin  { "userId": 1 }
GET    /api/users/all-emails

Without authentication, anyone can pretend to be anyone. They can delete other users' data, grant themselves admin access, or steal private information. The server has no way to tell the difference between a legitimate user and a malicious stranger.

Analogy

A building with no locks vs. a building with keycards. Without authentication, your server is a building where every door is wide open. Anyone can walk into any room — the CEO's office, the server room, the vault. Authentication is the keycard system. You prove who you are at the front desk, get a card, and only the doors you're authorized for will open.

What Authentication Actually Means

Authentication answers one question: "Who are you?"

When a request hits your server, the server needs to know:

  • Identity — Which user is making this request?
  • Proof — How do I know they are who they claim to be?
  • Trust — How do I remember them for future requests?

The rest of this course walks through exactly how each of those pieces works — from passwords to tokens to OAuth. By the end, you'll understand every step of the auth flow in a real application.

Passwords & Hashing

Lesson 2 of 8 — How the server checks your password without knowing it

The Naive Approach

The obvious way to handle passwords: store them in a database table.

-- DON'T DO THIS. Ever.
SELECT * FROM users;

| id | email              | password     |
|----|--------------------| -------------|
| 1  | alice@example.com  | hunter42     |
| 2  | bob@example.com    | p@ssw0rd     |
| 3  | carol@example.com  | letmein123   |

This is a catastrophe waiting to happen.

If anyone gains access to your database — a hacker, a rogue employee, a backup that leaks — they have every user's actual password. And since most people reuse passwords, those credentials likely work on their bank, email, and social accounts too.

The Solution: Hashing

Instead of storing the password, you store a hash of the password. A hash is a one-way mathematical function that turns any input into a fixed-length scrambled string. The critical property: you cannot reverse it.

How Login Works with Hashes

When a user creates an account, you hash their password and store the hash. When they log in, you hash the password they type and compare it to the stored hash:

Sign Up

1. User types "hunter42"

2. Server hashes it: $2b$10$Kx...

3. Server stores the hash in database

Log In

1. User types "hunter42"

2. Server hashes it: $2b$10$Kx...

3. Hashes match? You're in!

Salt: Preventing Rainbow Tables

There's a problem with plain hashing: if two users have the same password, they'll have the same hash. An attacker could precompute hashes for common passwords (a "rainbow table") and look them up.

The fix: salt. Before hashing, the server adds a random string (the salt) to your password. Each user gets a different salt, so even identical passwords produce different hashes.

// Without salt (bad) - same password = same hash
hash("password123") // always: 5f4dcc3b...

// With salt (good) - same password = different hash every time
hash("a9xK2m" + "password123") // ef92b778...
hash("q3Rv8p" + "password123") // 1a4f6c92...

bcrypt: The Industry Standard

In practice, you use bcrypt — a hashing algorithm designed specifically for passwords. It automatically handles salting, and it's intentionally slow (to make brute-force attacks impractical).

import bcrypt from 'bcrypt';

// Sign up: hash the password
const hash = await bcrypt.hash('hunter42', 10);
// hash = "$2b$10$Kx9Z8Yq3..." (includes salt + hash)

// Log in: compare password to stored hash
const isValid = await bcrypt.compare('hunter42', storedHash);
// isValid = true
Analogy

Hashing is like a paper shredder. You feed a document (your password) into the shredder and get a pile of confetti (the hash). You can always shred the same document and get the same pattern of confetti. But you cannot reassemble the confetti back into the original document. That's what makes it secure — even if someone steals all your hashes, they can't recover the passwords.

Sessions & Cookies

Lesson 3 of 8 — How the server remembers you after login

The Problem After Login

OK, so the user logs in with their email and password. The server checks the hash, and it matches. Great — the user is authenticated. But then they click to another page.

HTTP is stateless. Each request is independent. The server has no memory of the previous request. It's like having a conversation where you forget everything after each sentence.

So how does the server know that the next request comes from the same user who just logged in?

Sessions: A Temporary Pass

When the user logs in successfully, the server creates a session — a record that says "this user is currently logged in." The session gets a unique ID (a long random string).

// Server creates a session after successful login
const session = {
  id: "abc123xyz789",
  userId: 42,
  createdAt: "2026-06-18T10:00:00Z",
  expiresAt: "2026-06-19T10:00:00Z"
};

Cookies: Delivering the Pass

The server sends the session ID to the browser as a cookie. A cookie is a small piece of data that the browser stores and automatically includes in every future request to that server.

// Server response header
Set-Cookie: sessionId=abc123xyz789; HttpOnly; Secure; Path=/

The browser receives this, stores the cookie, and from now on, every time it makes a request to your server, it automatically attaches the cookie. The server reads the session ID from the cookie, looks it up, and knows who you are.

A wristband at a concert. You show your ticket at the gate (password). The staff verifies it and gives you a wristband (cookie). For the rest of the event, you just flash your wristband to get into any area — you don't need to show your ticket again. The wristband is proof that you already verified your identity.

Important Cookie Properties

  • HttpOnly — JavaScript cannot access the cookie (prevents XSS attacks from stealing it)
  • Secure — Cookie is only sent over HTTPS (prevents network eavesdropping)
  • SameSite — Cookie is only sent with requests from your own site (prevents CSRF attacks)
  • Expires / Max-Age — How long the cookie lasts (session cookies expire when you close the browser)

Tokens (JWT)

Lesson 4 of 8 — The token IS the proof

The Limitation of Sessions

Sessions work great, but they have a drawback: the server has to store every active session. If you have a million users logged in, that's a million sessions to keep track of. And if you have multiple servers, they all need access to the same session store.

JWT: JSON Web Tokens

A JWT is a different approach. Instead of the server storing session data, it encodes all the information into the token itself. The token contains your identity, and it's cryptographically signed so it can't be tampered with.

The server doesn't need to look anything up — it just reads the token and verifies the signature.

How JWT Signing Works

The signature is the key security feature. The server has a secret key that only it knows. It takes the header + payload, runs them through a signing algorithm with the secret key, and produces the signature.

Sessions vs. JWTs

Sessions

  • Server stores session data
  • Cookie contains only the session ID
  • Easy to invalidate (delete from server)
  • Requires a session store (database/Redis)

JWT

  • Server stores nothing
  • Token contains all user data
  • Can't be easily invalidated before expiry
  • Scales well across multiple servers
Analogy

A JWT is like a driver's license. It contains your information (name, address, date of birth) and has a hologram (signature) that proves it's real. A police officer doesn't need to call the DMV to verify you — they can read the license and check the hologram right there. But if someone peels off the hologram and sticks it on a fake ID, the hologram won't match, and the fraud is detected.

OAuth — "Login with Google"

Lesson 5 of 8 — How third-party login works

The Idea

You've seen those "Sign in with Google" or "Login with GitHub" buttons. They're convenient because you don't need to create a new password for every app. But what's actually happening behind the scenes?

The protocol that makes this work is called OAuth 2.0. It lets your app delegate authentication to a trusted provider (Google, GitHub, etc.) without ever seeing the user's password.

Step 1: The Redirect

When the user clicks "Sign in with Google," your app doesn't handle any passwords. Instead, it constructs a URL and redirects the browser to Google: 

GET https://accounts.google.com/oauth/authorize
  ?client_id=YOUR_APP_ID
  &redirect_uri=https://paperclip.dev/callback
  &scope=email profile
  &response_type=code
Analogy

A hotel giving you a room key authorized by the building owner. You (the user) want to check into a hotel (your app). Instead of the hotel verifying your identity directly, they call the building owner (Google) who already knows you. The building owner confirms your identity and tells the hotel "Yes, give them access." The hotel then gives you your own room key (session/JWT). You never showed the hotel your personal documents — the building owner vouched for you.

Why OAuth is Popular

  • No passwords to manage — Your app never touches user passwords
  • Users trust it — They already have Google/GitHub accounts
  • Scoped access — You only request what you need (email, profile, repos)
  • Revocable — Users can revoke access from Google's settings anytime

API Keys

Lesson 6 of 8 — How machines authenticate

Not Everything is a Human

So far we've talked about humans logging in with passwords or Google accounts. But what about automated systems, scripts, and agents? They can't type a password or click "Login with Google."

For machine-to-machine authentication, we use API keys: long, random secret strings that identify the caller.

What an API Key Looks Like

Click to reveal a sample API key:

Your Paperclip API Key:

pk_live_a8Kz9mR4xQ2vL7nB3wY5jF6hP0dC1eT8uI4oS

Never share this key or commit it to Git. Treat it like a password.

How API Keys Are Used

The key is included in the request headers. The server reads the header, looks up the key in its database, and determines who (or what) is making the request.

# Include API key in the Authorization header
GET /api/paperclips
Host: api.paperclip.dev
Authorization: Bearer pk_live_a8Kz9mR4xQ2vL7nB3wY5jF6hP0dC1eT8uI4oS
// Server-side: checking the API key
function authenticateApiKey(req, res, next) {
  const key = req.headers.authorization?.replace('Bearer ', '');

  const agent = await db.query(
    'SELECT * FROM api_keys WHERE key = $1',
    [key]
  );

  if (!agent) {
    return res.status(401).json({ error: 'Invalid API key' });
  }

  req.agent = agent;
  next();
}

A backstage pass. It doesn't have your photo or say who you are, but it proves you're allowed in. Anyone holding the pass gets access. API Keys vs. Passwords vs. JWTs

  • Passwords — for humans logging into a website
  • JWTs — for maintaining user sessions after login
  • API Keys — for machines, scripts, and agents calling your API

Authorization vs. Authentication

Lesson 7 of 8 — WHO you are vs. WHAT you're allowed to do

Two Different Questions

People often confuse these two concepts, but they're fundamentally different:

Authentication (AuthN)

WHO are you?

Verifying identity. Logging in. Proving you are who you claim to be.

Examples: password, fingerprint, Google OAuth, API key

Authorization (AuthZ)

WHAT can you do?

Checking permissions. Deciding what actions you're allowed to perform.

Examples: admin vs. viewer, read-only vs. read-write, scopes

Authentication always comes first. You can't determine what someone is allowed to do until you know who they are.

Roles and Permissions

Most apps organize authorization through roles. Each role has a set of permissions that define what actions are allowed.

const roles = {
  admin: {
    permissions: ['read', 'write', 'delete', 'manage_users', 'deploy']
  },
  editor: {
    permissions: ['read', 'write']
  },
  viewer: {
    permissions: ['read']
  }
};

// Middleware that checks permissions
function requirePermission(permission) {
  return (req, res, next) => {
    const userRole = roles[req.user.role];
    if (!userRole.permissions.includes(permission)) {
      return res.status(403).json({ error: 'Forbidden' });
    }
    next();
  };
}

Scopes (for APIs)

When you use OAuth, you request specific scopes — these are the permissions the user grants to your app. For example, when Paperclip uses the Google API:

// Paperclip only requests what it needs
scope="email profile"           // Read email and profile info
scope="email profile calendar"  // Also access Google Calendar

The user sees exactly what permissions you're requesting and can decide whether to allow it. Your app never gets more access than the user approves.

Auth in Practice

Lesson 8 of 8 — How Paperclip handles auth in the real codebase

The Full Picture

Now that you understand all the building blocks, let's see how they fit together in the Paperclip codebase. Paperclip uses three types of authentication for three types of users:

  • Board users (humans) — Log in with email/password or Google OAuth. Get a JWT.
  • Agents (AI) — Authenticate with API keys. Each agent has a unique key.
  • Runs (tasks) — Each agent run gets a short-lived JWT scoped to that specific task.

Step 1: Extract Token

The first thing the middleware does is look for credentials in the incoming request. It checks two places: the Authorization header (for API keys and bearer tokens) and cookies (for browser sessions). 

function extractToken(req) {
  // Check Authorization header first
  const authHeader = req.headers['authorization'];
  if (authHeader?.startsWith('Bearer ')) {
    return { type: 'bearer', token: authHeader.slice(7) };
  }

  // Fall back to cookie
  const sessionCookie = req.cookies?.['session'];
  if (sessionCookie) {
    return { type: 'session', token: sessionCookie };
  }

  return null;
}

Security Checklist

Here's what every developer should remember about auth. Click each item to expand:

1
Never store plain-text passwords
Always use bcrypt (or argon2) to hash passwords. Never use MD5 or SHA-256 alone — they're too fast and can be brute-forced. bcrypt is intentionally slow and includes salt automatically.
2
Use HTTPS everywhere
Without HTTPS, anyone on the network can read your cookies, tokens, and API keys in plain text. In the deployment course, you learned about SSL certificates — they're not optional for auth.
3
Set HttpOnly and Secure on cookies
HttpOnly prevents JavaScript from accessing the cookie (blocks XSS attacks). Secure ensures the cookie is only sent over HTTPS. SameSite prevents CSRF attacks. Always set all three.
4
Give tokens an expiration
JWTs should expire (usually in 15 minutes to 24 hours). If a token is stolen, the damage is limited to the expiration window. Use refresh tokens for longer sessions.
5
Never put API keys in your code
Use environment variables. In the deployment course, you learned about .env files and platform secrets. API keys committed to Git are the #1 cause of security breaches in open-source projects.
6
Apply the principle of least privilege
Give users and agents the minimum permissions they need. A viewer shouldn't be able to delete. An agent that reads data shouldn't have write access. If an account is compromised, the blast radius is minimized.

What You've Learned

  • Passwords are hashed with bcrypt — the server never stores the real password
  • Sessions & cookies keep you logged in across requests
  • JWTs are self-contained tokens with signed data
  • OAuth lets you delegate login to Google/GitHub
  • API keys authenticate machines and agents
  • Authentication is WHO, Authorization is WHAT
