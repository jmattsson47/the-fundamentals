Why React Exists

Lesson 1 of 8 — The problem React solves

The Problem: Updating the DOM by Hand

Before React, building interactive web pages meant manually finding elements in the HTML and updating them one by one. Every time your data changed, you had to figure out which parts of the page needed updating and write the code to do it.

This gets messy fast. Here's a simple example — updating a user's name on the page:

Imperative (vanilla JS)

// Find the element
const el = document.getElementById('name');

// Manually update it
el.innerHTML = 'Claude';

// Data changes? Find it again...
el.innerHTML = 'New name';

// More data changes? Again...
el.innerHTML = 'Another name';

// Now imagine 50 elements...

Declarative (React)

function Profile({ name }) {
  // Just describe WHAT it looks like
  return <h1>{name}</h1>;
}

// When name changes,
// React updates the page
// automatically.

// You never touch the DOM.

Analogy

Imperative is like telling a painter exactly which strokes to make — "Put blue here, then red there, then move that line 2 inches left." If anything changes, you have to give all new instructions. Declarative is like describing what you want the painting to look like — "I want a sunset over the ocean" — and the painter figures out the strokes. React is the painter.

React's Big Idea

React introduced a powerful concept: describe WHAT your UI should look like for any given data, and React figures out HOW to update the page.

  • You define the UI as a function of your data
  • React compares what's on screen to what should be on screen
  • React makes the minimum changes needed to update the DOM

This is why React is called a "declarative" UI library. You declare the result. React handles the process.

Quick Facts

  • React was created by Facebook (Meta) in 2013
  • It's the most popular UI library in the world
  • React only handles the view layer — what the user sees and interacts with
  • It works by building a "virtual" copy of the page and comparing it to the real page (more on this in Lesson 7)

Components

Lesson 2 of 8 — The building blocks of React

Everything is a Component

In React, everything you see on a page is built from components. A component is just a function that returns UI. That's it.

function Welcome() {
  return <h1>Hello, world!</h1>;
}

This function returns a piece of HTML (well, JSX — we'll cover that in the next lesson). React calls this function and puts the result on the page.

Components Compose

The real power of components is that they nest inside each other. A page is a component. It contains other components. Those contain other components. It's components all the way down.

function Header() {
  return <header><h1>Paperclip</h1></header>;
}

function AgentCard({ name }) {
  return <div>Agent: {name}</div>;
}

function AgentList() {
  return (
    <section>
      <AgentCard name="Claude" />
      <AgentCard name="Sasha" />
    </section>
  );
}

function App() {
  return (
    <div>
      <Header />
      <AgentList />
    </div>
  );
}

Analogy

Components are like LEGO bricks. Each brick is a small, reusable piece. You snap them together to build something bigger. An <AgentCard /> brick can be used on the list page, the detail page, or anywhere else you need it.

Rules of Components

  • Start with a capital letter — AgentCard, not agentCard. This is how React distinguishes components from regular HTML tags.
  • Return one root element — a component must return a single parent element (or use a fragment <>...</>).
  • Pure function — given the same inputs, a component should always return the same output.

JSX — HTML Inside JavaScript

Lesson 3 of 8 — The syntax that powers React

What is JSX?

JSX looks like HTML, but it lives inside JavaScript. It might feel weird at first, but it's incredibly powerful. It lets you mix your logic and your UI in the same place.

const element = <h1>Hello, world!</h1>;

This isn't a string. It isn't HTML. It's JSX — a syntax extension that gets compiled to regular JavaScript.

What JSX Compiles To

Under the hood, JSX is just syntactic sugar for function calls. Your build tool (like Vite or webpack) transforms it:

What you write (JSX)

<div className="card">
  <h2>{agent.name}</h2>
  <p>{agent.role}</p>
</div>

What the browser sees

React.createElement(
  'div',
  { className: 'card' },
  React.createElement(
    'h2', null, agent.name
  ),
  React.createElement(
    'p', null, agent.role
  )
)

You never have to write React.createElement yourself. JSX does it for you. But it's useful to know what's happening behind the curtain.

Dynamic Values with {Curly Braces}

The superpower of JSX is the curly braces {}. Any JavaScript expression inside them gets evaluated:

function AgentCard({ name, tasks }) {
  return (
    <div>
      <h2>{name}</h2>
      <p>Tasks: {tasks.length}</p>
      <p>{tasks.length > 5 ? 'Busy!' : 'Available'}</p>
    </div>
  );
}

Conditional Rendering

You can show or hide content based on conditions:

function Status({ isActive }) {
  return (
    <div>
      {isActive ? <span>Online</span> : <span>Offline</span>}
      {isActive && <span>Ready for tasks</span>}
    </div>
  );
}

Lists with .map()

To render a list of items, use .map() to transform data into JSX:

function AgentList({ agents }) {
  return (
    <ul>
      {agents.map(agent => (
        <li key={agent.id}>{agent.name}</li>
      ))}
    </ul>
  );
}

JSX Gotchas

  • class becomes className (because class is a reserved word in JS)
  • for becomes htmlFor (same reason)
  • Style is an object: style={{ color: 'red' }}
  • All tags must be closed: <img />, not <img>

Props — Passing Data Down

Lesson 4 of 8 — How components talk to each other

What are Props?

Props (short for "properties") are how a parent component sends data to a child component. They work exactly like function arguments.

// Parent passes data via attributes
<AgentCard name="Claude" role="CTO" status="active" />

// Child receives them as an object
function AgentCard({ name, role, status }) {
  return (
    <div className="card">
      <h3>{name}</h3>
      <p>{role}</p>
      <span>{status}</span>
    </div>
  );
}

Analogy

Props are like a mail slot. The parent component puts data in through the slot, and the child component reads what comes through. The child can't reach back through the slot to change what the parent sent — props are read-only. Data flows in one direction: parent to child.

Props Are Read-Only

A component must never modify its own props. This is a fundamental rule in React:

// WRONG - never do this!
function AgentCard(props) {
  props.name = "Hacked!"; // React will break
  return <h3>{props.name}</h3>;
}

// RIGHT - just read them
function AgentCard({ name }) {
  return <h3>{name}</h3>;
}

If a component needs to change data, it uses state — which is the next lesson.

Props Can Be Anything

  • Strings: name="Claude"
  • Numbers: count={42}
  • Booleans: isActive={true} or just isActive
  • Objects: agent={{ name: "Claude", role: "CTO" }}
  • Functions: onClick={() => alert("hi")}
  • Components: icon={<StarIcon />}

State — Data That Changes

Lesson 5 of 8 — Making components interactive

What is State?

Props come from the parent. But what about data that the component itself owns and changes? That's state. State is data that changes over time, and when it changes, React automatically re-renders the component.

import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  //     ^         ^              ^
  //     |         |              initial value
  //     |         function to update it
  //     current value

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Add 1
      </button>
    </div>
  );
}

The Re-render Cycle

  1. Event happens — user clicks, types, etc.
  2. setState is called — setCount(count + 1)
  3. React re-renders — calls your component function again
  4. New UI appears — React updates only what changed in the DOM
Analogy

State is like a whiteboard. You can erase it and write something new. Everyone looking at the whiteboard immediately sees the updated content. When you call setState, you're erasing and rewriting the whiteboard — and React makes sure everyone (every part of the UI) sees the new value.

Effects & Data Fetching

Lesson 6 of 8 — Side effects and talking to the outside world

What is a Side Effect?

Components are supposed to be pure — given the same props and state, return the same UI. But sometimes you need to reach outside the component: fetch data from an API, start a timer, or update the document title. These are side effects.

React gives you the useEffect hook for this:

import { useState, useEffect } from 'react';

function AgentList() {
  const [agents, setAgents] = useState([]);

  useEffect(() => {
    // This runs after the component renders
    fetch('/api/agents')
      .then(res => res.json())
      .then(data => setAgents(data));
  }, []); // empty array = run once on mount

  return (
    <ul>
      {agents.map(a => <li key={a.id}>{a.name}</li>)}
    </ul>
  );
}

Remember fetch from Course 3 — APIs & HTTP? useEffect is where you put it in React.

The Dependency Array

The second argument to useEffect controls when the effect runs:

  • useEffect(fn) — runs after every render (rarely what you want)
  • useEffect(fn, []) — runs once on mount (like "page load")
  • useEffect(fn, [value]) — runs when value changes

Cleanup Functions

Effects can return a cleanup function that runs when the component unmounts or before the effect re-runs:

useEffect(() => {
  const timer = setInterval(() => {
    console.log('tick');
  }, 1000);

  // Cleanup: runs when component unmounts
  return () => clearInterval(timer);
}, []);

Fetching Data Pattern

Here's the complete pattern for fetching and displaying data — one of the most common things you'll do in React:

function AgentDetail({ agentId }) {
  const [agent, setAgent] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(`/api/agents/${agentId}`)
      .then(res => res.json())
      .then(data => {
        setAgent(data);
        setLoading(false);
      })
      .catch(err => {
        setError(err.message);
        setLoading(false);
      });
  }, [agentId]); // re-fetch when agentId changes

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;

  return <h1>{agent.name}</h1>;
}

The React Lifecycle

Lesson 7 of 8 — How React decides what to update

Mount, Update, Unmount

Every component goes through three phases:

  • Mount — the component appears on screen for the first time. React creates the DOM nodes and runs initial effects.
  • Update — the component re-renders because its props or state changed. React compares old and new, and updates only what changed.
  • Unmount — the component is removed from the screen. React runs cleanup functions and removes DOM nodes.

The Virtual DOM

React doesn't update the real DOM directly. Instead, it works with a lightweight copy called the Virtual DOM:

  1. When state changes, React builds a new Virtual DOM tree
  2. React compares the new tree to the old tree (called "diffing")
  3. React figures out the minimum set of changes needed
  4. React applies only those changes to the real DOM

This is why React is fast. Instead of replacing everything, it surgically updates only the parts that changed.

OLD VIRTUAL DOM
<div>
  <h1>Claude</h1>
  <p>Tasks: 3</p>
  <span>active</span>
</div>
→
NEW VIRTUAL DOM
<div>
  <h1>Claude</h1>
  <p>Tasks: 4</p>
  <span>active</span>
</div>

React sees only one thing changed (3 to 4) and updates just that text node.

Why Keys Matter in Lists

When rendering lists, React needs a way to track which items are which. That's what the key prop is for:

// BAD - using index as key
{agents.map((agent, index) => (
  <AgentCard key={index} agent={agent} />
))}

// GOOD - using unique ID as key
{agents.map(agent => (
  <AgentCard key={agent.id} agent={agent} />
))}

Interactive: Keys in Action

Watch what happens when we shuffle a list. With proper keys, React moves items. Without keys, it rebuilds from scratch:

With proper keys, React knows each item by its ID and just moves them in the DOM instead of re-creating them.

Routing: App.tsx

Paperclip's main App.tsx maps URLs to page components. When the URL changes, React renders the matching page:

// App.tsx - simplified
import { BrowserRouter, Routes, Route } from 'react-router-dom';

function App() {
  return (
    <BrowserRouter>
      <Layout>
        <Routes>
          <Route path="/" element={<Dashboard />} />
          <Route path="/agents" element={<AgentList />} />
          <Route path="/agents/:id" element={<AgentDetail />} />
          <Route path="/tasks" element={<TaskBoard />} />
        </Routes>
      </Layout>
    </BrowserRouter>
  );
}

Context: Sharing Data Across Components

Some data needs to be available everywhere — like the current company info. React Context lets you share data without passing props through every level:

// CompanyContext.tsx
const CompanyContext = createContext(null);

function CompanyProvider({ children }) {
  const [company, setCompany] = useState(null);

  useEffect(() => {
    fetch('/api/company')
      .then(res => res.json())
      .then(setCompany);
  }, []);

  return (
    <CompanyContext.Provider value={company}>
      {children}
    </CompanyContext.Provider>
  );
}

// Any component, anywhere in the tree:
function Header() {
  const company = useContext(CompanyContext);
  return <h1>{company?.name}</h1>;
}

Custom Hooks: useQuery

Paperclip uses custom hooks to encapsulate common patterns. The useQuery pattern wraps fetching, loading, and error states into one reusable function:

// useQuery.ts - simplified
function useQuery(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(url)
      .then(res => res.json())
      .then(d => { setData(d); setLoading(false); })
      .catch(e => { setError(e); setLoading(false); });
  }, [url]);

  return { data, loading, error };
}

// Usage in any component:
function AgentList() {
  const { data: agents, loading, error } = useQuery('/api/agents');

  if (loading) return <Spinner />;
  if (error) return <ErrorMessage error={error} />;

  return agents.map(a => <AgentCard key={a.id} agent={a} />);
}

Putting It All Together

Here's a bird's-eye view of how all the React concepts connect in Paperclip:

  • Components — every page and UI element is a component (AgentCard, TaskBoard, Layout)
  • Props — parent components pass data down (<AgentCard agent={agent} />)
  • State — components manage their own data (useState for form inputs, toggles, filters)
  • Effects — data fetching on mount and when IDs change (useEffect with useQuery)
  • Context — shared data like company info available everywhere (CompanyContext)
  • Routing — URL-to-page mapping in App.tsx (react-router-dom)

The File Structure

  • client/src/App.tsx Root component with routes
  • client/src/components/ Reusable UI components
  • client/src/pages/ Page-level components (Dashboard, AgentList, etc.)
  • client/src/hooks/ Custom hooks (useQuery, useAuth, etc.)
  • client/src/context/ React Context providers
  • client/src/lib/ Utility functions, API helpers

What You Now Know

  • React solves the problem of keeping the UI in sync with data
  • Components are functions that return UI — they compose like building blocks
  • JSX is HTML-in-JavaScript that compiles to React.createElement calls
  • Props pass data down from parent to child (one-way, read-only)
  • State is data that changes over time and triggers re-renders
  • Effects handle side effects like fetching data and timers
  • The Virtual DOM makes updates efficient by diffing and patching
  • Real apps combine all of these — routing, context, hooks, and components
