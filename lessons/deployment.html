Deployment

localhost vs The World

Lesson 1 of 8 — Why nobody else can see your app (yet)

When you ran pnpm dev for Paperclip and opened localhost:3100 in your browser, you saw your app working. That felt like magic. But here is the thing: nobody else can see it.

Try sending that link to a friend. Go ahead, text them "localhost:3100." They will get nothing. A blank page. An error. Why?

What is "localhost"?

localhost is a special name that means "this computer, right here." Every computer has it. When your browser goes to localhost:3100, it is talking to software running on your own machine. It never touches the internet.

It is like talking to yourself. Useful for practice, but nobody else hears you.

Analogy

localhost = cooking dinner at home. You can eat it. Your family at the table can eat it. But nobody walking down the street can taste your food. Deployment = opening a restaurant. Now anyone with the address can walk in and eat. That is what we are learning to do.

What does "deployment" actually mean?

Deployment is the process of taking code from your computer and putting it on a server that anyone in the world can reach. That is it. The whole concept in one sentence.

💻
Your Laptop

Only you can access it. Turns off when you close the lid.

$ pnpm dev
ready - started server on localhost:3100
# only YOUR browser can open this

URL: http://localhost:3100 — works on your machine only

🌎
A Server (Vercel)

Anyone on earth can access it. Runs 24/7. Never sleeps.

$ vercel deploy
Deploying...
Ready! https://paperclip-abc123.vercel.app
# anyone with this URL can open it

URL: https://paperclip-abc123.vercel.app — works for everyone, everywhere

The gap we need to bridge

Right now your Paperclip code lives on your laptop. By the end of this tutorial, you will understand every step between "code on your laptop" and "app anyone can visit." Click through each question below.

Where does the code go? ▶

Your code gets uploaded to a server in the cloud — a computer in a data center run by a company like Vercel. We will cover this in Lesson 4.

How does the code run there? ▶

The server runs a "build" step (compiling, bundling) and then serves the output. Think of it like pnpm dev but for production. Lesson 3 covers this.

How does someone find it? ▶

Through a URL and DNS — the system that turns names like "google.com" into the actual address of the server. Lesson 5 dives deep into this.

What is a Server?

Lesson 2 of 8 — Just a computer that never sleeps

The word "server" sounds fancy and technical. It is not. A server is just a computer that serves things to other computers. When you go to google.com, your browser (the "client") asks Google's server for the webpage, and the server sends it back.

Your laptop could technically be a server. But it would be a terrible one. Here is why:

💻 Your Laptop as a Server

  • ❌ Turns off when you close the lid
  • ❌ Disconnects when you leave WiFi
  • ❌ IP address changes every network
  • ❌ Firewall blocks incoming requests
  • ❌ Gets slow when you open Photoshop
  • ❌ One user overloads it

☁️ Cloud Server (Vercel, AWS)

  • ✔ Runs 24 hours, 365 days a year
  • ✔ Connected to high-speed backbone
  • ✔ Fixed public IP address
  • ✔ Configured to accept all requests
  • ✔ Dedicated to serving your app
  • ✔ Scales to handle thousands of users
Analogy

Your laptop is like your home kitchen. You can cook there, but it is not set up for 200 guests. A cloud server is like a commercial kitchen in a restaurant. It is built to handle volume, it is always open, and it has a fixed address people can find.

What does a server actually do?

Click each role to learn more:

1. It listens for requests

A server sits and waits. It has a port open (like port 3100 in your Paperclip app, or port 443 for HTTPS). When someone types your URL, the request arrives at that port.

// Your Paperclip server does this under the hood:
const server = createServer(app);
server.listen(3100, () => {
  console.log('Listening on port 3100');
});

2. It processes the request

When a request comes in, the server figures out what is being asked for. "They want the homepage? The API for paperclips? An image?" It runs your code to generate the right response.

// Request comes in: GET /api/paperclips
if (path === '/api/paperclips') {
  const clips = await db.query('SELECT * FROM paperclips');
  // got the data, now respond...
}

3. It sends a response

The server sends back HTML, JSON, images, or whatever was requested. Your browser takes that response and renders it on screen. The whole trip — request to response — takes milliseconds.

// Server sends back JSON:
res.json({
  paperclips: [
    { id: 1, name: "Standard Silver", count: 42 }
  ]
});

4. It stores data

Servers often connect to a database (like the SQLite database in your Paperclip app). The database lives on the server too (or on another specialized server). This is where your paperclip inventory persists, even when users close their browsers.

// The database file on your server:
/data/paperclip.db ← SQLite file

// In production, often a managed database:
postgres://db.vercel.com:5432/paperclip

The Build Step

Lesson 3 of 8 — Turning raw code into something fast

Here is something that surprises people: the code you write is not the code that gets deployed. Not exactly. Your source code goes through a build step first — a process that transforms it into optimized, production-ready code.

Analogy

Raw ingredients are not the finished dish. You have flour, eggs, sugar, and butter. The build step is the baking. What comes out of the oven is a cake — smaller, combined, and ready to eat. You would not serve someone raw flour.

What the build step does

Click each task to see what it means:

TypeScript → JavaScript ▶

Browsers do not understand TypeScript. The build step compiles your .tsx and .ts files into plain JavaScript that any browser can run.

// Before (TypeScript):
const getClips = (count: number): Paperclip[] => { ... }

// After (JavaScript):
const getClips = (count) => { ... }

The type annotations are stripped away. They were useful for you while coding, but the browser does not need them.

Bundling — many files become few ▶

Your project has dozens of files that import each other. The bundler traces all the imports and combines them into a few optimized files. Fewer files = fewer network requests = faster loading.

// You wrote 47 files:
src/components/Header.tsx
src/components/ClipCard.tsx
src/utils/format.ts
... 44 more files

// Bundler outputs 2 files:
dist/index.js (everything combined)
dist/chunk-vendor.js (libraries)
Minification — removing whitespace and shortening names ▶

The minifier removes all spaces, newlines, and comments, and renames variables to single letters. The code does the same thing but takes up way less space.

// Before (readable):
function calculateTotal(paperclips) {
  const total = paperclips.reduce((sum, clip) => {
    return sum + clip.price;
  }, 0);
  return total;
}

// After (minified):
function c(a){return a.reduce((s,e)=>s+e.price,0)}

Same logic. Same result. 60% less data to download.

Tree Shaking — removing unused code ▶

If you import a library but only use 2 of its 50 functions, the build step throws away the other 48. This is called "tree shaking" — shaking the tree and letting the dead branches fall.

// You imported:
import { format, parse } from 'date-fns';

// date-fns has 200+ functions
// Tree shaking keeps ONLY format and parse
// Everything else is removed from the bundle

The build command

For your Paperclip project, the build is triggered by a simple command:

$ pnpm build

info - Creating an optimized production build...
info - Compiled successfully
info - Collecting page data...
info - Generating static pages
Done in 8.2s
Key insight: When you run pnpm dev, there is no build step — your code runs directly for speed. When you deploy, the build step runs first to make everything small and fast for real users.

How Vercel Works

The Deploy Flow

Vercel gives you a unique URL like paperclip-abc123.vercel.app. Anyone can visit it. Your app is live. The whole process takes about 30-60 seconds.

Two ways to deploy

Manual deploy with the CLI

You run vercel deploy in your terminal. Good for testing. You are in full control of when it goes live.

# Deploy a preview (test URL):
$ vercel deploy

# Deploy to production (the real URL):
$ vercel deploy --prod

Automatic deploy with Git

Connect your GitHub repo to Vercel, and every time you git push, Vercel automatically deploys. This is how most teams work — push code, it goes live automatically.

# You just push your code like normal:
$ git add .
$ git commit -m "add paperclip sorting"
$ git push origin main

# Vercel detects the push and deploys automatically
# No extra commands needed!
This is the most common workflow: You write code, push to GitHub, and Vercel handles the rest. You do not even need the Vercel CLI for this.

DNS — How Names Become Addresses

Lesson 5 of 8 — The phone book of the internet

When you type google.com into your browser, your computer does not actually know where Google is. Computers communicate using IP addresses — numbers like 142.250.80.46. But nobody wants to memorize numbers.

That is what DNS does. DNS (Domain Name System) converts human-friendly names into computer-friendly numbers.

Analogy

DNS is the phone book of the internet. You know your friend's name (google.com), but to call them you need their phone number (142.250.80.46). DNS looks up the name and gives you the number.

Try a DNS Lookup

Type a domain name and see what happens:

The Full DNS Journey

When you type a URL and press Enter, here is what happens in about 50 milliseconds:

1

Browser checks its cache

"Have I looked this up recently?" If yes, it already knows the IP. Done. If not, keep going.

2

Ask the operating system

Your Mac or PC has its own cache. Maybe another app already looked this up. Still nothing? Keep going.

3

Ask your ISP's DNS server

Your internet provider (Comcast, AT&T, etc.) runs DNS servers. They handle millions of lookups and cache popular ones. If they have it, great.

4

Ask the root DNS servers

If nobody has the answer cached, the query goes to the root DNS servers — 13 server clusters that know where to find information about every domain extension (.com, .org, .io, etc.).

5

Get the IP address

The authoritative DNS server for that domain responds with the IP address. Your browser now knows where to connect. The answer gets cached so next time it is instant.

How this connects to Vercel

When Vercel gives you a URL like paperclip-k4f2j.vercel.app, they also set up DNS records that point that name to their servers' IP addresses. If you add a custom domain like paperclip.com, you would add DNS records pointing it to Vercel.

DNS Records for Your App

Type Name Value What it does
A paperclip.com 76.76.21.21 Points domain to Vercel's IP
CNAME www cname.vercel-dns.com Points www subdomain to Vercel
What is the difference between an A record and a CNAME? ▶

A record = "this name points to this IP address." Direct mapping.

CNAME record = "this name points to another name." It is like a redirect. www.paperclip.com points to cname.vercel-dns.com, which then resolves to the actual IP.

Think of an A record as writing someone's home address directly. A CNAME is like writing "same address as my friend John" — it points to John, and John's address is the real answer.

CDN — Copies Everywhere

Lesson 6 of 8 — Why Vercel is fast from anywhere in the world

Imagine if Google had only one server, sitting in a closet in California. Every single person on Earth — Tokyo, London, Sydney, Sao Paulo — would have to send their request across the ocean to that one closet and wait for a response to travel all the way back.

That would be slow. Distance creates delay. Light in a fiber optic cable takes about 70 milliseconds to cross the Atlantic Ocean. That is just the physics; you cannot make it faster.

The solution: do not have one server. Have copies everywhere.

Analogy

A CDN is like a chain restaurant. McDonald's does not cook all its burgers in one kitchen in Chicago and ship them worldwide. They have kitchens in every city. When you order in Tokyo, the Tokyo kitchen makes your burger. A CDN does the same thing with your website.

What is a CDN?

CDN = Content Delivery Network. A network of servers spread across the world, each holding a copy of your website. When a user requests your site, the nearest server responds.

Without CDN vs With CDN

😟 Without CDN (single server)

  • Server in Virginia, USA
  • User in Tokyo: ~200ms latency
  • User in Sydney: ~250ms latency
  • User in London: ~100ms latency
  • Server overloaded under load
  • If server goes down, everything is down

⚡ With CDN (Vercel Edge)

  • Servers in 23+ regions worldwide
  • User in Tokyo: ~10ms latency
  • User in Sydney: ~15ms latency
  • User in London: ~8ms latency
  • Load spread across all servers
  • If one goes down, others take over
This is why you chose Vercel: When you deploy Paperclip, it does not go to one server. It goes to edge servers worldwide. Every user gets a fast experience, no matter where they are. You get this for free.

Environments

Lesson 7 of 8 — Development, staging, and production

You would not test a parachute by jumping out of a plane. You would test it on the ground first, then from a low height, then from the real thing. Software works the same way. You have different environments for different stages of confidence.

The Three Environments

Click each environment to learn about it:

💻
Development (your laptop)

Where you write code and try things out

This is what you have been doing. Running pnpm dev on localhost:3100. You can break things, try crazy ideas, and nobody sees it. It is your personal sandbox.

# You run this every day:
$ pnpm dev
http://localhost:3100 ← only you see this

Database: Local SQLite file on your laptop
API keys: Test keys that hit sandbox APIs
Who sees it: Only you

🔎
Staging (preview deployment)

A dress rehearsal before going live

A staging environment is a copy of production for testing. On Vercel, every push to a non-main branch creates a preview deployment — a real URL anyone can visit, but it is not the "official" production URL.

# Push to a branch (not main):
$ git push origin feature/sorting

# Vercel creates a preview URL:
https://paperclip-feature-sorting-k4f2j.vercel.app
↑ real URL, but not production

Database: Test database (copy of production structure, fake data)
API keys: Test keys, same as development
Who sees it: You and your team for testing

🌎
Production (live for everyone)

The real deal. Real users. Real data.

This is the live version that real users see. Breaking production means your users see errors. That is why you test in development and staging first.

# Push to main branch:
$ git push origin main

# Vercel deploys to production:
https://paperclip.vercel.app ← THE real URL
# (or your custom domain: paperclip.com)

Database: Production database with real user data
API keys: Live keys that process real payments
Who sees it: Everyone on the internet

Environment Variables: Why .env Files Matter

Your app needs different settings in different environments. The database URL in development is not the same as in production. Environment variables are how you handle this.

# .env (local development)
DATABASE_URL="file:./dev.db"
API_KEY="sk_test_abc123_not_real"
NEXT_PUBLIC_URL="http://localhost:3100"
NODE_ENV="development"

Uses a local SQLite file. Test API key. localhost URL.

# .env.staging (preview deployments)
DATABASE_URL="postgres://staging-db.vercel.com:5432/paperclip_test"
API_KEY="sk_test_xyz789_not_real"
NEXT_PUBLIC_URL="https://paperclip-preview.vercel.app"
NODE_ENV="production"

Uses a test database in the cloud. Still test API keys. Preview URL.

# .env.production (live site)
DATABASE_URL="postgres://prod-db.vercel.com:5432/paperclip"
API_KEY="sk_live_REAL_KEY_KEEP_SECRET"
NEXT_PUBLIC_URL="https://paperclip.vercel.app"
NODE_ENV="production"

Real database. LIVE API key (charges real money!). Production URL.

Key rule

Never commit .env files to Git. They contain secrets (API keys, database passwords). Your .gitignore file should list .env so Git does not track it. On Vercel, you set environment variables in the dashboard instead.

Every Step Explained

Step 1 You push code to GitHub ▶

You have been writing code for Paperclip. You added a feature, tested it locally on localhost:3100, and it works. You commit and push:

$ git add .
$ git commit -m "add paperclip color filter"
$ git push origin main

Your code is now on GitHub's servers. This is not deployment yet — your code is stored, but nobody can visit it as a website.

Step 2 Vercel detects the push ▶

Because you connected your GitHub repo to Vercel (linked to jared-2093), Vercel has a "webhook" — a notification system. GitHub tells Vercel: "Hey, new code was pushed to main."

Vercel wakes up and says: "New code? Time to deploy." It pulls down your latest code from GitHub.

Step 3 Vercel runs the build ▶

Vercel's servers run your build command. For Paperclip, that is pnpm build. This compiles TypeScript, bundles your React components, minifies everything, and tree-shakes unused code. (Everything from Lesson 3.)

Your 180 MB of source code becomes ~48 KB of optimized output. If the build fails (syntax error, missing import), the deploy stops and you get an error message.

Step 4 Output is deployed to edge servers ▶

The build output is copied to Vercel's edge network — servers in 23+ cities worldwide. (Everything from Lesson 6.) Within seconds, your app is available from Tokyo, London, Sao Paulo, everywhere.

Vercel gives you a URL: https://paperclip.vercel.app

Step 5 A user types the URL, DNS routes them ▶

Someone in Tokyo types paperclip.vercel.app in their browser. DNS kicks in (Lesson 5): the domain name gets resolved to an IP address. Vercel's smart DNS routes them to the nearest edge server — the one in Tokyo.

Step 6 The page loads ▶

The Tokyo edge server sends back your optimized HTML, CSS, and JavaScript. The browser renders Paperclip. The user sees your app. They have no idea about build steps, edge networks, or DNS lookups. They just see a fast, working app.

Total time from git push to live: about 30-60 seconds.
