Why Test?

Lesson 1 of 8 — The case for automated testing

The Problem: Fix One Bug, Create Two More

You fix the login page. Ship it. Then someone reports that the signup page is broken. You fix signup. Now the dashboard is broken. Every change feels like playing whack-a-mole — you can never be sure that fixing one thing didn't break three others.

This is called a regression — when something that used to work stops working because of a new change. Regressions are the most common type of bug in real software.

Without Tests: Fear of Changing Code

The developer's inner monologue without tests:

  • "I need to refactor this function, but what if I break something?"
  • "This code is messy, but I'm afraid to touch it."
  • "I think this change is safe... let me just deploy and pray."
  • "We should update this dependency, but who knows what it'll break."

Fear makes codebases rot. When developers are afraid to change code, the code gets worse over time.

With Tests: Confidence

The developer's inner monologue with tests:

  • "I'll refactor this — if I break anything, the tests will tell me."
  • "Let me clean up this messy code. The tests have my back."
  • "I'll make this change, run the tests, and know immediately if it's safe."
  • "Let's update that dependency — we'll know in seconds if anything broke."

Tests give you the confidence to move fast without breaking things.

See It In Action

Here's a simple function that calculates a discount. Watch what happens when we change the code — the test catches the bug instantly.

Function
function calculateDiscount(price, percent) {
  return price * (percent / 100);
}
Test
test('calculates 20% of $100 as $20', () => {
  expect(calculateDiscount(100, 20)).toBe(20);
});
✓ calculates 20% of $100 as $20

Tests: 1 passed, 1 total
Time: 0.03s
Analogy

Tests are like a safety net for tightrope walkers. The net doesn't help you walk better, but it lets you be bold. You can try daring moves because you know the net will catch you if you fall. Without the net, you inch along in fear. With it, you can dance.

What a Test Actually Does

A test is just code that checks other code. It does three things:

  1. Arrange — Set up the inputs
  2. Act — Call the function
  3. Assert — Check the result is what you expected

That's it. Every test follows this pattern: set up, do something, check the result.

Unit Tests

Lesson 2 of 8 — Testing one function at a time

What Is a Unit Test?

A unit test tests a single function in isolation. You give it inputs and check that the output is correct. No database, no network, no UI — just the function and its logic.

Think of it like testing a single brick before building a wall. You want to make sure each brick is solid on its own.

The Anatomy of a Test

Paperclip uses Vitest — a modern, fast testing framework. Here's what a test looks like:

import { describe, it, expect } from 'vitest';
import { calculateBudgetRemaining } from './budget';

describe('calculateBudgetRemaining', () => {
  it('returns remaining budget after spending', () => {
    const result = calculateBudgetRemaining(1000, 350);
    expect(result).toBe(650);
  });

  it('returns zero when budget is fully spent', () => {
    const result = calculateBudgetRemaining(500, 500);
    expect(result).toBe(0);
  });

  it('never returns negative (clamps to zero)', () => {
    const result = calculateBudgetRemaining(100, 200);
    expect(result).toBe(0);
  });
});

Let's break down the key parts:

  • describe — Groups related tests together (like a folder)
  • it — Defines a single test case (reads like "it should...")
  • expect — Makes an assertion: "I expect this value to be..."
  • .toBe() — The matcher: checks if the value equals what you expect

Try It: Pass or Fail?

Here's the calculateBudgetRemaining function. Modify it and see if the tests pass or fail.

budget.ts
export function calculateBudgetRemaining(budget: number, spent: number): number {
  const remaining = budget - spent;
  return Math.max(0, remaining);
}
✓ returns remaining budget after spending
✓ returns zero when budget is fully spent
✓ never returns negative (clamps to zero)

Tests: 3 passed, 3 total
Time: 0.02s

Common Matchers

Vitest (and Jest) give you many ways to check values:

expect(value).toBe(42);              // exact equality
expect(value).toEqual({ name: 'Jared' }); // deep equality (objects)
expect(value).toBeTruthy();           // not null/undefined/false/0
expect(value).toBeNull();             // is null
expect(value).toContain('hello');     // string or array contains
expect(value).toThrow();              // function throws an error
expect(value).toHaveLength(3);        // array/string length
Key Idea

Unit tests are fast — really fast. A typical project runs hundreds of unit tests in under a second. That speed is crucial: you want to run tests every time you save a file and get instant feedback.

What to Test (and What Not To)

Lesson 3 of 8 — Focusing your testing effort

Test Behavior, Not Implementation

A common mistake: testing how code works internally instead of what it does. If you test implementation details, your tests break every time you refactor — even when the behavior is the same.

Bad: Testing implementation

// This test breaks if you rename the internal variable
expect(calculator._internalSum).toBe(42);

Good: Testing behavior

// This test survives refactoring — it only checks the output
expect(calculator.getTotal()).toBe(42);

What NOT to Test

Testing everything is a waste of time. Skip these:

  • Framework code — Don't test that React renders a component. React's own tests handle that.
  • Trivial getters/setters — getName() { return this.name; } doesn't need a test.
  • Third-party libraries — Don't test that lodash.sortBy works. Trust the library.
  • Exact CSS/UI layout — "Is this button exactly 42px wide?" is a fragile, low-value test.

What TO Test

  • Business logic — Calculations, rules, validations. This is where bugs have the biggest impact.
  • Edge cases — What happens with zero? With null? With a huge number? With an empty string?
  • Public API contracts — "Given this input, does this function return the right output?"
  • Error handling — Does the code fail gracefully when things go wrong?
  • Integration points — Where your code talks to other systems (database, APIs).
Rule of Thumb

Test the things that would embarrass you if they broke in production. If a bug in a function would cause users to see wrong data, lose money, or be unable to log in — that function needs tests.

Integration Tests

Lesson 4 of 8 — Testing pieces working together

Beyond Single Functions

Unit tests check individual functions. But what about when those functions work together? An integration test tests multiple pieces of your system as a group: send an HTTP request, hit the route handler, which calls the service, which talks to the database, and check the response.

A Paperclip Example

Let's test the endpoint that creates a new issue. This test exercises the full stack: HTTP layer, route handler, service logic, and database.

import { describe, it, expect } from 'vitest';
import { app } from '../src/app';

describe('POST /api/issues', () => {
  it('creates a task and returns it', async () => {
    const response = await app.request('/api/issues', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        title: 'Fix search bar',
        status: 'todo',
        agentId: 'agent-1',
      }),
    });

    expect(response.status).toBe(201);

    const task = await response.json();
    expect(task.title).toBe('Fix search bar');
    expect(task.status).toBe('todo');
    expect(task.id).toBeDefined();
  });
});

The test acts like a client — it sends a real HTTP POST request to /api/issues with a JSON body containing title, status, and agentId. The test framework creates an in-memory server, so no actual network call is made — but the server processes it exactly like a real request.

Why Integration Tests Matter

Unit tests verify each brick is solid. Integration tests verify the bricks fit together. You could have perfect bricks that don't connect — integration tests catch those gaps.

  • Does the route actually call the right service function?
  • Does the database query return data in the format the API expects?
  • Does error handling work end-to-end (bad input to error response)?
Analogy

Unit tests are like testing each instrument in an orchestra individually. Integration tests are like a rehearsal — you check that the violins and cellos sound good together, that the timing is right, that the conductor's cues actually work.

End-to-End Tests

Lesson 5 of 8 — Testing from the user's perspective

The Full Picture

An end-to-end (E2E) test tests your entire application from the user's perspective. It opens a real browser, clicks real buttons, fills in real forms, and verifies what actually appears on screen.

Paperclip uses Playwright — a tool that automates browser actions. You write code that says "go to this page, click this button, type in this field, check that this text appears."

A Playwright Test

import { test, expect } from '@playwright/test';

test('can create a new agent', async ({ page }) => {
  // Navigate to the agents page
  await page.goto('http://localhost:3000/agents');

  // Click the "New Agent" button
  await page.click('button:has-text("New Agent")');

  // Fill in the form
  await page.fill('input[name="name"]', 'Research Bot');
  await page.fill('input[name="budget"]', '500');
  await page.selectOption('select[name="model"]', 'claude-sonnet');

  // Submit the form
  await page.click('button:has-text("Create Agent")');

  // Verify the new agent appears on the page
  await expect(page.locator('text=Research Bot')).toBeVisible();
});

E2E Tests Are Slow but Powerful

  • Pro: Tests exactly what users experience — if the E2E test passes, the feature works for real users
  • Pro: Catches bugs that unit and integration tests miss (CSS hiding a button, broken navigation)
  • Con: Slow — each test takes seconds, not milliseconds
  • Con: Flaky — timing issues can cause tests to randomly fail
  • Con: Expensive to maintain — UI changes break E2E tests

This is why the testing pyramid says to write few E2E tests — just enough to cover the critical user journeys.

Mocking & Fixtures

Lesson 6 of 8 — Faking the things you can't (or don't want to) use for real

The Problem: Real Dependencies Are Messy

When you test a function that calls the database, you have a choice: use the real database, or fake it. Real databases are slow, require setup, and can have leftover data from other tests. External APIs cost money and might be down. Sometimes faking is better.

What Is a Mock?

A mock is a fake version of something real. Instead of calling the actual database, your test calls a mock that instantly returns pre-defined data. Same interface, no side effects.

See the Difference

Click to switch between a test with a real DB call and the same test using a mock:

Test (Real Database)
it('fetches agent by id', async () => {
  // Arrange: insert real data into the DB
  await db.insert(agents).values({
    id: 'agent-1',
    name: 'Research Bot',
    budget: 500,
  });

  // Act
  const result = await getAgent('agent-1');

  // Assert
  expect(result.name).toBe('Research Bot');
});
Characteristics

Slow: Has to connect to a real database

Setup: Requires a running DB instance

Cleanup: Need to delete test data afterward

Realistic: Tests the real database behavior

Fixtures: Pre-Built Test Data

A fixture is pre-built test data you reuse across tests. Instead of creating test data from scratch in every test, you define it once:

// fixtures/agents.ts
export const testAgent = {
  id: 'agent-1',
  name: 'Research Bot',
  budget: 500,
  model: 'claude-sonnet',
  status: 'active',
};

export const testIssue = {
  id: 'issue-1',
  title: 'Fix search bar',
  status: 'todo',
  agentId: 'agent-1',
};

// Then in your tests:
it('assigns issue to agent', () => {
  const result = assignIssue(testIssue, testAgent);
  expect(result.agentId).toBe('agent-1');
});
Analogy

Mocking is like a flight simulator. You're testing the pilot's skills without risking a real plane. The controls feel real, the instruments look real, but it's all simulated. The pilot gets realistic practice without the danger, expense, or unpredictability of actual flight.

When to Mock, When Not To

Mock when:

  • The real dependency is slow (database, network, file system)
  • The real dependency has side effects (sending emails, charging cards)
  • The real dependency is unreliable (external APIs that might be down)
  • You want to test error scenarios (simulate a database crash)

Don't mock when:

  • The real thing is fast and reliable — just use it
  • You're writing integration tests — the whole point is to test real connections
  • The mock would be more complex than the real thing

Test-Driven Development (TDD)

Lesson 7 of 8 — Write the test first, then the code

The TDD Cycle

TDD flips the normal order: instead of writing code then testing it, you write the test first. Then you write just enough code to make the test pass. Then you clean up.

Red

Write a failing test

→

Green

Make it pass

→

Refactor

Clean up

Walk Through an Example

Let's build a function called formatAgentStatus using TDD. Click each phase to see the cycle in action:

Test File: formatAgentStatus.test.ts
// Step 1: Write the test FIRST — before the function exists

import { describe, it, expect } from 'vitest';
import { formatAgentStatus } from './formatAgentStatus';

describe('formatAgentStatus', () => {
  it('returns "Active" with green icon for active agents', () => {
    expect(formatAgentStatus('active')).toEqual({
      label: 'Active',
      color: 'green',
    });
  });

  it('returns "Paused" with yellow icon for paused agents', () => {
    expect(formatAgentStatus('paused')).toEqual({
      label: 'Paused',
      color: 'yellow',
    });
  });

  it('returns "Error" with red icon for errored agents', () => {
    expect(formatAgentStatus('error')).toEqual({
      label: 'Error',
      color: 'red',
    });
  });
});
Tests have not been run yet. Click "2. Red: Run It" to see what happens.

Phase: RED

We write the test before the function exists. The test describes the behavior we want: given a status string, return a label and a color. Notice how the test reads like a specification — anyone can understand what formatAgentStatus should do just by reading the tests.

Why TDD Works

  • Forces you to think first. Writing the test first means you have to decide what the function should do before you write it.
  • Guarantees test coverage. Every function starts with a test — you can't forget to test it later.
  • Keeps code simple. You only write enough code to pass the test. No over-engineering.
  • Instant feedback. You know within seconds if your code works.
Key Idea

TDD doesn't mean you must always write tests first. In practice, most developers use TDD selectively — for complex business logic, tricky algorithms, or bug fixes (write a test that reproduces the bug, then fix it). The important thing is that the code ends up with good tests, regardless of the order you wrote them.

Testing in Practice

Lesson 8 of 8 — How Paperclip actually tests its code

Paperclip's Test Setup

Paperclip uses two tools, each for a different kind of test:

Vitest

Unit & Integration Tests

  • Fast — runs in milliseconds
  • Tests functions, services, API routes
  • Runs in Node.js (no browser needed)

Playwright

End-to-End Tests

  • Slower — runs in seconds
  • Tests the full app in a real browser
  • Simulates real user actions

The Test Commands

These are the commands you'll use every day:

# Run all unit and integration tests once
pnpm test

# Run tests in watch mode — re-runs on every file save
pnpm test:watch

# Run end-to-end tests
pnpm test:e2e

# Run a specific test file
pnpm test src/services/budget.test.ts

# Run tests matching a pattern
pnpm test --grep "budget"

The one you'll use most is pnpm test:watch. It sits in the background and re-runs relevant tests every time you save a file. You get feedback in under a second.

CI/CD: Tests on Every Pull Request

Remember Course 6 (How Git Works) and Course 5 (How Deployment Works)? Tests connect to both:

What happens when you open a Pull Request:

  1. You push your branch to GitHub
  2. CI (Continuous Integration) automatically runs all tests
  3. GitHub shows a green checkmark or red X on your PR
  4. If tests fail, you can't merge — this protects the main branch
  5. If tests pass, reviewers can merge with confidence
# .github/workflows/test.yml — Runs on every PR
name: Tests
on: [pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pnpm install
      - run: pnpm test
      - run: pnpm test:e2e

Paperclip's File Structure

The Testing Mindset

  • Write tests as you go, not at the end. Tests written after the fact feel like a chore; tests written alongside the code feel like a tool.
  • Test the happy path first, then edge cases. Make sure the normal flow works before worrying about errors.
  • A failing test is a gift. It tells you exactly what's broken and where — before your users find out.
  • 100% coverage is not the goal. Covering the critical paths with good tests is more valuable than covering everything with shallow tests.
