Why Git Exists

Lesson 1 of 8 — The problem Git solves

The Problem

Imagine you're building your app. You have a file called app.js. It works. Then you make a change and... everything breaks. You want to go back, but you already saved over the old version.

Without Git, people solve this the worst way possible:

The "file naming" disaster:

app.js
app_v2.js
app_v3_fixed.js
app_FINAL.js
app_FINAL_v2.js
app_FINAL_FINAL_USE_THIS_ONE.js

Click to add more versions:

This is chaos. Which version actually works? Which one has the login fix? Which one has the new homepage? Nobody knows. And this is just one file. Real projects have hundreds of files.

The Solution: Git

Git tracks every change to every file automatically. Instead of copying files, you tell Git: "Save a snapshot of my project right now." Git remembers the entire state of your project at that moment. You can jump back to any snapshot at any time.

With Git, you just have:

app.js

One file. Git remembers all the versions for you behind the scenes.

Click each version to see what Git remembers:

Analogy

Git is like an unlimited undo button that remembers every version of every file you've ever saved — and lets you jump to any point in time. Think of it like a time machine for your code.

Some Quick Facts

  • Git was created in 2005 by Linus Torvalds (the same person who created Linux)
  • Almost every software project in the world uses Git
  • Git works on your computer — it doesn't need the internet
  • GitHub is a website that stores Git projects online (more on that in Lesson 6)

Commits — Saving Checkpoints

Lesson 2 of 8 — Snapshots of your project

What is a Commit?

A commit is a snapshot of your entire project at one moment in time. Every time you make a commit, Git saves:

  • A unique ID (hash) — a long string like a3f7b2c that identifies this exact snapshot
  • A message — a note you write describing what changed, like "Fixed login bug"
  • Who made it — the author's name and email
  • When — the exact date and time

What a Commit Looks Like in Code

When you look at a commit, Git shows you exactly what changed — lines that were added (in green) and lines that were removed (in red):

Before (commit e4f5g6h)
function login(user, pass) {
  if (user === "admin") {
    return "Welcome!";
  }
}
After (commit i7j8k9l)
function login(user, pass) {
  if (user === "admin" && pass === hash(pass)) {
    return "Welcome!";
  }
  return "Invalid credentials";
}
Analogy

Commits are like save points in a video game. Every time you reach a milestone, you save. If you mess up later, you can always reload from any save point. The difference is Git never runs out of save slots.

The Three Areas

Lesson 3 of 8 — How changes become commits

The Three Stages of Git

When you make changes, they don't become a commit automatically. Changes flow through three areas:

Working Directory
Your files as you see them
index.html
style.css
app.js
→
Staging Area
git add — "include these"
→
Repository
git commit — "saved!"

Try it! Move files through the three areas:

Why Have a Staging Area?

You might wonder: why not just save everything at once? The staging area lets you be selective. Maybe you changed 5 files, but only 3 are ready. You stage just those 3, commit them, and keep working on the other 2.

The commands:

git add style.css          # Stage one file
git add .                  # Stage ALL changed files
git commit -m "Fixed styles"  # Save the staged files as a commit
Analogy

Think of it like packing a suitcase.

Working directory = clothes scattered around your room. Staging area = clothes you've put in the suitcase. Commit = zipping the suitcase shut. You can pick and choose which clothes go in before you zip it.

Branches — Parallel Universes

Lesson 4 of 8 — Working on features without breaking things

What is a Branch?

A branch is a separate line of development. Think of it as a parallel universe where you can try things out without affecting the main code.

Every Git project starts with one branch, usually called main. When you want to work on something new — like adding a login page — you create a new branch. You make your changes there. When everything works, you merge it back into main.

Interactive Branch Diagram

Click the buttons to see branches in action:

This is the main branch. All commits are in a straight line. This is your production code — the real, working version of your project.

Why Use Branches?

  • Safety — If your experiment breaks, main is untouched
  • Teamwork — Multiple people can work on different features at the same time
  • Organization — Each branch has a clear purpose: "add-login", "fix-bug-123", "redesign-header"

Branch Commands

git branch my-feature       # Create a new branch
git checkout my-feature     # Switch to that branch
git checkout -b my-feature # Create AND switch (shortcut)
git branch                   # List all branches
git checkout main           # Switch back to main
Analogy

Branches are like parallel universes in a movie. In one universe, you try adding a new feature. In another universe (main), everything stays the same. If the feature works, you merge the universes together. If it doesn't, you just throw away the branch universe — main was never touched.

Merging & Conflicts

Lesson 5 of 8 — When branches collide

What is Merging?

When your feature branch is done, you merge it into main. Git takes all the changes from your branch and combines them with main.

Most of the time, this works perfectly. Git is smart enough to combine changes from different files, or even different parts of the same file.

git checkout main           # Switch to main
git merge my-feature        # Bring changes from my-feature into main

But Sometimes... Conflicts

A merge conflict happens when two branches change the same line of the same file. Git doesn't know which version you want, so it asks you to decide.

What a Conflict Looks Like

Git marks the conflict right in the file with special markers:

function getGreeting(user) {
<<<<<<< main
  return "Hello, " + user.name;
=======
  return `Welcome back, ${user.firstName}!`;
>>>>>>> feature-branch
}

What the markers mean:

  • <<<<<<< main — this is what the code looks like in main
  • ======= — divider between the two versions
  • >>>>>>> feature-branch — this is what the code looks like in the feature branch

Resolve the Conflict

You have three options. Click one to see the result:

After Resolving

Once you've picked the right version, you:

git add greeting.js        # Mark the conflict as resolved
git commit -m "Merged feature-branch, resolved conflict"
Analogy

It's like two people editing the same paragraph in a Google Doc at the same time. Google Docs can handle it if you're editing different paragraphs. But if you both change the same sentence, someone needs to decide which version to keep (or combine them).

How to Avoid Conflicts

  • Pull often — regularly download the latest changes from main
  • Small branches — the longer a branch lives, the more likely conflicts are
  • Communicate — if two people are working on the same file, talk about it

Remote & GitHub

Lesson 6 of 8 — Your code in the cloud

Local vs Remote

So far, everything we've talked about happens on your computer. That's your local repository. But what if your computer crashes? What if someone else needs to work on the same project?

That's where remotes come in. A remote is a copy of your repository stored somewhere else — usually on GitHub.

The Relationship

The Key Commands

git push                   # Upload your commits to GitHub
git pull                   # Download others' commits from GitHub
git clone url              # Copy an entire repository from GitHub

Clone — You Already Did This!

Remember this command?

gh repo clone paperclipai/paperclip

This used Git behind the scenes to clone the Paperclip repository from GitHub to your computer. It downloaded every file, every commit, and the full history — a complete copy of the project.

The Full Picture

Here's how the pieces fit together:

  • clone = download the whole project for the first time
  • push = upload your new commits to GitHub
  • pull = download new commits others have pushed
  • fork = make your own copy of someone else's project on GitHub
Analogy

GitHub is like Google Drive for code. You work on files on your computer (local), and you sync them to the cloud (remote). Push is like uploading, pull is like downloading. Clone is like downloading a shared folder for the first time.

Pull Requests

Lesson 7 of 8 — The collaboration workflow

Why Not Push Directly to Main?

In a real team, you never push directly to the main branch. Why? Because main is the "live" code. If you push something broken, everyone is affected.

Instead, teams use Pull Requests (PRs). A Pull Request says: "Hey, I made some changes on my branch. Can someone review them before they go into main?"

Git in Practice

Lesson 8 of 8 — Your daily workflow

The Daily Commands

Here are the commands you'll use every single day, in order. Click each one to learn more:

1
git status
What changed?

This is the command you'll run the most. It shows you:

  • Which files you've changed
  • Which files are staged (ready to commit)
  • Which files are new and untracked
$ git status
On branch main
Changes not staged for commit:
  modified:   src/app.js
Untracked files:
  src/search.js

Think of it as asking Git: "What's going on right now?"

2
git add
Stage your changes

Tell Git which files you want to include in your next commit.

git add src/search.js      # Stage one specific file
git add src/               # Stage everything in a folder
git add .                  # Stage everything that changed

This doesn't save anything yet — it just puts files in the "ready to commit" pile.

3
git commit -m "message"
Save a checkpoint

Create a snapshot of all your staged changes. The message should describe what you changed and why.

git commit -m "Add search bar to dashboard"

Good commit messages:

  • Good: "Fix login timeout by increasing session duration"
  • Good: "Add search feature to dashboard"
  • Bad: "Fixed stuff"
  • Bad: "asdfasdf"
4
git push
Upload to GitHub

Send your commits to GitHub so they're backed up and your team can see them.

git push                   # Push to the current branch's remote
git push origin main       # Be specific about where
git push -u origin my-branch # First push of a new branch
5
git pull
Download the latest

Download any new commits that others have pushed to GitHub. Always do this before starting new work.

git pull                   # Download and merge latest changes

This is like hitting "refresh" on a Google Doc to get everyone else's changes.

6
git branch / git checkout
Switch contexts

Create branches and switch between them.

git branch                 # List all branches
git checkout -b new-feature # Create and switch
git checkout main           # Switch to main

Pro tip: newer versions of Git also use git switch instead of git checkout — both work.

The Typical Workflow

Here's what a developer does every day:

# Morning: get the latest code
git pull

# Start working on a feature
git checkout -b add-notifications

# ... write code for a while ...

# Check what you changed
git status

# Stage the changes
git add .

# Save a checkpoint
git commit -m "Add notification bell to header"

# Upload to GitHub
git push -u origin add-notifications

# Open a Pull Request on GitHub
# Get it reviewed and merged

Connecting Back to Paperclip

When you ran:

gh repo clone paperclipai/paperclip

Git downloaded:

  • Every file in the project — thousands of them

In one command, you got the entire project and its complete history. That's the power of Git.
