Lesson 1: Tables

The building blocks of every database.

You already know this part. A database table is rows and columns — just like a spreadsheet.

Each column is a type of information (name, email, role). Each row is one record — one person, one task, one agent.

Analogy: Think of a table as a filing cabinet drawer. Every folder in the drawer has the same tabs (Name, Date, Status). Each folder is one record. The drawer label is the table name.

The question is: how do tables talk to each other? That's where it gets interesting.

Lesson 2: Primary Keys & Foreign Keys

How every row gets an identity, and how tables point to each other.

Primary Key (PK) — unique ID for this row Foreign Key (FK) — points to a row in another table

Every row needs a unique identifier — something that says "this is the one and only row #3." That's the Primary Key (PK). Usually it's just a number that auto-increments: 1, 2, 3, 4...

Analogy: A primary key is like a Social Security Number. Two people can have the same name, but they can never have the same SSN. It's how you tell them apart, guaranteed.

Now, when one table needs to reference a row in another table, it stores that row's primary key. This reference is called a Foreign Key (FK).

Lesson 3: Relationships

One-to-many, many-to-many, and why it matters.

Tables don't exist in isolation. The whole power of a database is in how tables relate to each other. There are three main types:

The most common. One agent has many tasks. One company has many agents. One project has many issues.

Analogy: One parent, many children. A parent can have 5 kids, but each kid has exactly one biological mother. The "many" side (tasks) stores the foreign key pointing back to the "one" side (agent).

The interesting one. An agent can have many skills, and a skill can belong to many agents.

Lesson 4: The Schema

The blueprint that defines your entire database.

You asked what a "schema" is. It's the blueprint — the definition of every table, every column, every relationship, and every rule, before any data goes in.

Analogy: If the database is a building, the schema is the architectural plans. It defines how many rooms there are, how big they are, which doors connect which rooms, and what each room is for. You draw the plans first, then fill the building with people (data).

Here's what a schema looks like in code (SQL). This is what those 102 Paperclip migrations are doing — building this blueprint piece by piece:

CREATE TABLE agents ( id INTEGER PRIMARY KEY, -- unique ID, auto-increments name TEXT NOT NULL, -- required, can't be blank role TEXT, -- optional adapter TEXT NOT NULL, -- what kind of agent budget DECIMAL(10,2), -- dollars and cents company_id INTEGER REFERENCES companies(id) -- FK! );

Let's break down what each piece means:

▢ Schema Vocabulary
TermWhat it means
CREATE TABLEMake a new table with this name
INTEGER, TEXTColumn types — what kind of data goes here (number, text, date, etc.)
PRIMARY KEYThis column uniquely identifies each row
NOT NULLThis column is required — you can't leave it blank
REFERENCESThis column is a foreign key — it points to a row in another table
DECIMAL(10,2)A number with up to 10 digits and 2 decimal places

When Paperclip says it has 102 migrations, that means 102 schema changes applied in order. Migration 1 might create the companies table. Migration 2 adds agents. Migration 50 might add a budget column to agents. Each migration is a small, ordered change to the blueprint.

Analogy: Migrations are like building permits. Permit #1: build the foundation. Permit #23: add a second bathroom. Permit #51: knock down a wall and expand the kitchen. You apply them in order, and at the end you have the current building.

Lesson 5: Queries

Asking your database questions.

Once you have tables with data, you need to ask questions. That's what SQL queries do. SQL stands for Structured Query Language — it's how you talk to a database.

Analogy: If the database is a librarian with perfect memory, a query is the question you ask them. "Show me all books by Stephen King published after 2020, sorted by rating." The librarian knows exactly where to look.

Lesson 6: Paperclip's Database

Now you can appreciate what 102 migrations really means.

Everything you just learned — tables, keys, relationships, schemas, queries — Paperclip uses all of it. Here's a simplified map of how the major tables connect:

companies | ├── agents (many agents per company) | ├── heartbeats (wake-up schedule per agent) | ├── agent_skills (many-to-many with skills) | └── cost_events (every dollar spent, tracked) | ├── projects (many projects per company) | └── goals (many goals per project) | └── issues (many tasks per goal) | ├── comments | ├── attachments | ├── work_products | └── execution_logs | ├── budgets (spending limits per scope) ├── approvals (human sign-off records) ├── activity_log (audit trail of everything) ├── secrets (encrypted credentials) ├── routines (scheduled recurring work) └── plugins (installed extensions)

Every arrow you see is a foreign key. Every indentation is a one-to-many relationship. The whole tree starts from companies — that's why Paperclip can run multiple companies in one database with complete isolation.

You now understand the core concepts behind every database in the world — from your bank to Instagram to Paperclip. Tables, keys, relationships, schemas, queries, and migrations. Everything else is just details built on top of these fundamentals.

What you learned:

Table = a spreadsheet of one kind of thing
Primary Key = each row's unique ID
Foreign Key = a column that points to another table's row
Schema = the blueprint defining all tables and rules
Query = a question you ask the database
Migration = one ordered change to the blueprint
